# A-12.5: Transaction Boundary Review(A-13 착수 전)

> **목적**: A-13 구현 전에 Financial Workflow의 Transaction Boundary를 확정하기 위한 설계를 검토한다. 구현/DB/테스트 변경 없음.
> **기준 문서**: ADR-003(Financial Events), ADR-006(Staff Earning Policy), `docs/A12_STAFF_EARNING_ARCHITECTURE.md`, `docs/A12_IMPLEMENTATION_READY.md`
> 작성일: 2026-06-26

---

## PART 1 — 현재 Financial Workflow 분석(실제 코드 기준)

`closeSession()`(`lib/features/session/data/session_repository.dart` 246~325행)을 그대로 읨으면, 현재 5단계 중 **2단계는 이미 구현, 3단계는 미구현(예정)**이다.

| 단계 | 실제 코드 | 구현 상태 | 분류 |
|---|---|---|---|
| 1. Session 상태 변경 | `_db.update(_db.paymentSessions)...write(status: 'closed', endAt: now)` (309~313행, 워크플로의 **마지막**에 실행됨) | ✅ 구현됨 | **State Transition** |
| 2. Settlement 계산 | 결제수단 합계 검증(`paidTotal == session.finalAmount`) + `_db.batch()`로 `paymentMethodBreakdowns` insert(269~284행) | ✅ 구현됨 | **Snapshot 생성**(검증은 순수 계산이지만, `PaymentMethodBreakdown` 저장은 이벤트 기록 — ADR-002와 같은 결의 "품목/결제 단위 사실 기록") |
| 3. `StaffEarningLedger` 생성 | `StaffEarningEngine.calcEarnings()` 호출 후 `_db.batch()`로 `staffEarningLedgers` insert(286~303행) | ✅ 구현됨(A-12) | **Snapshot 생성**(ADR-006 — Event 아님, 파생·불변 스냅샷) |
| 4. Receipt 처리 | 없음 | ❌ 예정 | **외부 연동**(프린터/PDF 등 DB 트랜잭션 밖의 부수효과) |
| 5. Sync 처리 | 없음(이 앱은 오프라인 우선 단일 인스턴스 — README.md 원칙) | ❌ 예정(현재는 동기화 대상 자체가 없음) | **외부 연동**(네트워크 I/O) |

### 중요한 관찰 — 현재 순서와 트랜잭션 경계

코드 순서는 **(2) Settlement 저장 → (3) Ledger 저장 → (1) 상태 변경**이다(상태 변경이 워크플로의 처음이 아니라 **마지막**에 일어난다 — `status`가 `'closed'`로 바뀌기 전까지는 이미 결제 내역과 Ledger가 먼저 DB에 쓰여 있는 상태가 한순간 존재한다).

**현재 이 3단계는 하나의 트랜잭션으로 묶여 있지 않다** — `_db.batch()`(결제수단), `_db.batch()`(Ledger, 조건부), `_db.update()`(상태 변경)가 **각각 독립적으로 실행되고 즉시 커밋**된다(같은 함수 안에서 순차적으로 `await`될 뿐, `_db.transaction()`으로 감싸여 있지 않음). 이는 §PART3의 실패 시나리오 분석에서 핵심 전제가 된다.

## PART 2 — Transaction Boundary 후보 비교

### ① 하나의 트랜잭션(Session 상태 변경 + Settlement + Ledger 생성)

| 기준 | 내용 |
|---|---|
| 장점 | 세 단계가 전부 성공하거나 전부 실패한다 — "결제는 기록됐는데 마감 상태는 안 바뀐" 같은 중간 상태가 원천적으로 불가능. 데이터 일관성 추론이 가장 단순함(원자성 보장). |
| 단점 | 트랜잭션이 길어질수록 잠금 시간이 늘어남(다만 SQLite 단일 인스턴스·로컬 트랜잭션이라 이 비용은 미미). Receipt/Sync가 나중에 이 트랜잭션에 합류하려는 유혹이 생길 수 있는데, 그건 ②/③에서 피하는 것이 맞음(외부 연동을 DB 트랜잭션에 끌어들이면 트랜잭션이 외부 시스템 응답시간에 종속됨). |
| 데이터 일관성 | **가장 높음** — 세 테이블(`payment_sessions`, `payment_method_breakdowns`, `staff_earning_ledgers`)이 항상 같은 상태(전부 기록됨 또는 전부 없음)로 유지됨. |
| 장애 복구 | 단순함 — 트랜잭션이 중간에 끊기면 SQLite가 자동으로 전체 rollback하므로, 재시도 시 "어디까지 됐었는지" 따질 필요가 없음(처음부터 다시 `closeSession()` 호출하면 됨, 단 PART5의 멱등성 보장이 필요). |
| 오프라인 적합성 | 높음 — 로컬 SQLite 트랜잭션이라 네트워크와 무관, 오프라인 우선 원칙과 가장 잘 맞음. |
| 구현 난이도 | 낮음 — Drift의 `_db.transaction()`으로 현재 3개의 독립 호출을 감싸기만 하면 됨(구현 자체는 단순, 다만 이번 단계에서 구현하지 않음). |

### ② 상태 변경 + Ledger만 하나의 트랜잭션(Receipt/Sync는 별도 처리)

| 기준 | 내용 |
|---|---|
| 장점 | "마감 = 상태 변경 + 수당 확정"이라는 핵심 사업 로직만 원자성을 갖고, Receipt/Sync 같은 외부 연동의 불확실성(프린터 응답 지연, 네트워크 단절)이 핵심 트랜잭션을 막지 않음. |
| 단점 | Settlement(결제수단 저장)가 어느 쪽에 속하는지 애매함 — ①이 "Settlement도 포함"이라면 ②는 Settlement을 빼는 건지 포함하는 건지 재정의가 필요(아래 §PART2 "결론" 참조). |
| 데이터 일관성 | 핵심 금전 데이터(상태+수당)는 일관성 보장, Receipt/Sync는 "있어도 좋고 나중에 보정해도 되는" 부수 데이터로 격리. |
| 장애 복구 | Receipt/Sync 실패가 "마감 자체의 실패"가 아니므로, 별도의 재시도/보정 절차만 있으면 됨(마감을 되돌릴 필요 없음). |
| 오프라인 적합성 | 높음 — 핵심 트랜잭션은 ①과 동일하게 로컬, Sync는 원래부터 별도 처리(이 앱은 현재 Sync 자체가 없음). |
| 구현 난이도 | 낮음~중간 — 핵심 트랜잭션은 ①과 동일하게 단순하나, Receipt/Sync를 "트랜잭션 밖"으로 명확히 분리하는 호출 순서 설계가 추가로 필요. |

### ③ 모든 단계를 각각 독립 트랜잭션(현재 상태와 가장 유사)

| 기준 | 내용 |
|---|---|
| 장점 | 각 단계가 서로의 실패에 영향받지 않음 — 구현이 제일 단순(현재 코드가 이미 이 모양에 가깝다). |
| 단점 | **PART3에서 분석하는 모든 부분 실패 시나리오가 가능해진다** — "결제는 기록됐는데 상태는 여전히 open" 같은 불일치가 실제로 발생할 수 있음. 이 불일치를 감지/복구하는 별도 로직이 필요해짐(현재는 그 로직이 없음). |
| 데이터 일관성 | **가장 낮음** — 단계 수만큼 부분 실패 조합이 늘어남. |
| 장애 복구 | 복잡함 — "어느 단계까지 성공했는지"를 매번 조회해서 판단해야 재시도가 가능함(현재 코드는 이 판단 로직이 없음 — 재호출하면 `BusinessRuleException`이 던져질 뿐, 안전하게 "이어서 진행"하지 못함). |
| 오프라인 적합성 | 보통 — 트랜잭션 자체는 로컬이라 오프라인과 무관하지만, 일관성 문제는 오프라인 여부와 무관하게 발생. |
| 구현 난이도 | 낮음(현재 상태와 유사하므로 추가 작업이 거의 없음) — 그러나 "낮은 구현 난이도"가 "낮은 운영 리스크"를 의미하지 않음. |

### 권장안(확정하지 않음)

**②(상태 변경 + Ledger만 하나의 트랜잭션, Settlement은 그 트랜잭션에 포함)을 1차 권장으로 제시한다 — 단, 이번 단계에서 확정하지 않는다.**

근거: Settlement(`paymentMethodBreakdowns` 저장)은 "결제수단 검증을 통과했다"는 사실 자체가 상태 전환의 전제조건이므로, 상태 변경과 분리되면 "결제 기록은 있는데 마감은 안 됐다"는 어색한 상태가 생긴다 — 따라서 Settlement은 사실상 ①의 "하나의 트랜잭션" 안에 있어야 자연스럽다. Receipt/Sync는 DB 트랜잭션 경계 밖의 외부 연동(§PART1 분류)이므로 애초에 트랜잭션에 들어갈 후보가 아니다. **결과적으로 ①과 ②는 실질적으로 같은 경계를 가리키게 된다**(Settlement까지 포함한 "핵심 금전 트랜잭션" 하나 + Receipt/Sync는 그 밖). 이 일치 자체가 "권장"이지 "확정"은 아니다 — A-13에서 Receipt/Sync의 구체적 구현 방식이 정해지면 경계가 정말 이렇게 그려지는지 재확인이 필요하다.

## PART 3 — 실패 시나리오 분석

> 전제: 현재 코드는 ③(독립 트랜잭션)에 가깝다(PART1) — 아래 분석은 **현재 코드 기준**으로 실제로 발생 가능한 상태를 짚는다.

| 시나리오 | 재시도 가능한가 | Rollback 필요한가 | 데이터 복구 가능한가 |
|---|---|---|---|
| **Session CLOSED 성공 / Ledger 실패** | **현재 코드에서는 발생 불가능** — Ledger 생성(286~303행)이 상태 변경(309~313행)보다 먼저 실행되므로, Ledger가 실패하면 예외가 던져져 상태 변경 자체가 시도되지 않는다. (다만 이 시나리오 이름 자체가 "상태 변경이 워크플로 처음"이라는 가정 하의 것이므로, §PART1에서 짚은 "현재 순서는 Settlement→Ledger→상태변경"이라는 사실과 대조해 볼 필요가 있다.) | 해당 없음(발생 안 함) | 해당 없음 |
| **Ledger 성공 / Receipt 실패**(Receipt 도입 후) | **재시도 가능해야 한다** — Receipt는 DB 트랜잭션 밖 외부 연동(PART1)이므로, Ledger가 이미 확정된 뒤 Receipt만 다시 시도하는 것이 합리적이다. | 불필요 — Ledger를 되돌릴 이유가 없다(이미 확정된 Snapshot, ADR-006). | 가능 — Ledger 데이터로부터 Receipt를 다시 생성할 수 있어야 한다(Receipt가 Ledger/SessionItem의 파생물이라는 전제가 필요, A-13에서 확정). |
| **Receipt 성공 / Sync 실패**(Sync 도입 후) | **재시도 가능해야 한다** — Sync는 가장 바깥쪽 외부 연동이며, 로컬 데이터(Session/Ledger/Receipt)는 이미 확정돼 있으므로 Sync만 별도로 재시도하면 된다. | 불필요 | 가능 — 로컬에 모든 데이터가 남아있으므로 Sync는 "다시 보내기"만 하면 됨. |
| **앱 종료**(closeSession() 실행 도중) | 단계별로 다름 — DB 쓰기가 커밋된 단계까지는 보존되고, 그 이후 단계는 미실행 상태로 남는다. **현재 코드(독립 트랜잭션)에서는 "결제수단은 저장됐는데 Ledger도 상태변경도 안 된" 상태가 가능하다.** | 부분적으로 필요 — 미완료 워크플로를 감지해 이어서 진행하거나 되돌리는 정책이 A-13에서 필요. | 가능(SQLite 파일은 그대로 남음) — 단, "어디까지 됐는지" 판단 로직이 없으면 복구가 수동이 됨. |
| **전원 종료**(쓰기 도중) | SQLite의 저널링(WAL/rollback journal)에 의해 **이미 커밋된 트랜잭션은 보존, 커밋 전 트랜잭션은 자동 rollback**된다(SQLite 자체의 ACID 보장 — Drift/SQLite 차원에서 이미 해결된 문제). 다만 "독립적인 여러 트랜잭션 중 일부만 커밋된 상태"는 여전히 가능(PART1의 핵심 리스크). | 자동(SQLite가 처리) — 그러나 "독립 트랜잭션 여러 개" 구조에서는 이미 커밋된 앞 단계는 rollback되지 않는다. | 가능(파일 손상이 없다면) |
| **DB Write 실패**(디스크 풀, 권한 오류 등) | 케이스에 따라 다름 — 일시적 오류(락 경합)는 재시도 가능, 영구적 오류(디스크 풀)는 사용자 개입 필요. | 자동(SQLite가 실패한 단일 트랜잭션은 즉시 rollback) | 실패한 단계 자체는 기록되지 않으므로 복구할 "손상된 데이터"는 없음 — 다만 그 이전에 커밋된 독립 트랜잭션은 그대로 남아 §PART2의 일관성 문제와 동일하게 작용. |

**종합**: 현재(③에 가까운) 구조에서 가장 현실적인 위험은 "**앱/전원 종료 또는 중간 단계 실패로 일부만 커밋되는 것**"이다 — 이는 PART2의 ①/②가 해결하려는 정확한 문제이며, 본 PART의 분석이 PART2의 권장 방향을 다시 한번 뒷받침한다.

## PART 4 — Workflow 책임 검토(구현하지 않음, 책임 정의만)

```
Workflow(이름/형태 미정 — A12_IMPLEMENTATION_READY.md PART2에서 이미 "UseCase/Application
Service/Command Handler 중 무엇인지는 A-13 이후 결정"으로 명시)
  │
  ├─▶ SessionRepository   : Session/SessionItem 상태 조회·전이(상태 변경 자체의 영속화)
  ├─▶ SettlementEngine    : 결제수단 합계와 finalAmount 일치 검증(이미 closeSession() 내부 로직으로 존재)
  ├─▶ StaffEarningEngine  : SessionItem으로부터 직원 수익 계산(순수 계산, A-12에서 이미 구현)
  ├─▶ ReceiptService(예정): 확정된 Session/Item/Ledger로부터 영수증 표현(출력/PDF 등) 생성
  └─▶ SyncService(예정)   : 확정된 데이터를 외부(클라우드 등)로 전송 — 현재 앱에는 대상 자체가 없음
```

각 컴포넌트의 책임만 정의한다(구현 안 함):

- **Workflow**: 위 5개 컴포넌트를 "어떤 순서로, 어떤 트랜잭션 경계로" 호출할지 조율한다. 계산도 하지 않고 DB도 직접 만지지 않는다 — 호출 순서/경계 결정만.
- **`SessionRepository`**: Session/SessionItem 테이블 CRUD + (현재는 임시로 겸하는) 상태 전이 절차. A12_IMPLEMENTATION_READY.md PART2에서 이미 "임시 책임"으로 정의됨 — 본 PART는 그 정의를 재사용할 뿐, 새로 결정하지 않는다.
- **`SettlementEngine`**: 결제수단 합계 vs `finalAmount` 검증(순수 판정 로직). 현재는 `closeSession()` 내부에 인라인으로 존재 — 별도 클래스로 분리할지는 A-13의 "Workflow 계층 도입 여부"와 함께 결정될 사안(본 PART는 책임만 명명, 분리 여부는 결정하지 않음).
- **`StaffEarningEngine`**: 이미 A-12에서 구현됨 — 변경 없음.
- **`ReceiptService`(가칭)**: 확정된 데이터로부터 사람이 읽을 영수증 표현을 만든다. DB에 쓰지 않는다(읍기 전용 + 외부 출력).
- **`SyncService`(가칭)**: 확정된 데이터를 외부 시스템에 전달한다. 현재 앱 구조(오프라인 우선 단일 인스턴스)에는 대상이 없어 A-13 시점에도 "본격 구현"이 아니라 "후크 자리만 정의"하는 수준일 가능성이 높다.

**본 PART는 이름과 책임 경계만 제안한다 — 클래스를 만들거나 코드를 옮기지 않는다.**

## PART 5 — Idempotency 검토

`closeSession()`은 이미 "두 번째 호출은 `BusinessRuleException`으로 거부된다"는 가드를 갖고 있다(`session.status != open` 체크). 이것만으로 PART5가 요구하는 3가지가 전부 안전한지 따로 확인한다.

### Ledger 중복 생성

**현재 코드 기준으로는 안전하다.** Ledger 생성(286~303행)은 상태 변경(309~313행)보다 **먼저** 일어나므로, 첫 호출이 Ledger까지 성공하고 상태 변경 직전에 실패하면 — 그 세션은 여전히 `status='open'`이라 **재호출이 가능**하고, 재호출 시 Ledger가 또 한 번 insert돼 **중복 생성된다.** 이건 현재 구조의 실제 위험이다(PART3의 "앱 종료" 행과 직결).

**전략 제안(구현 안 함)**: Ledger insert 전에 "이 세션에 이미 Ledger가 있는가"를 확인하고 있으면 skip하는 가드, 또는 PART2 ①/②의 단일 트랜잭션으로 묶어 "Ledger는 성공했는데 상태변경은 실패"라는 중간 상태 자체를 없애는 방법 — 후자가 더 근본적인 해법이다(부분 실패가 발생할 수 없으면 멱등성 가드도 덜 필요해짐).

### Receipt 중복 생성

**Receipt가 외부 부수효과(프린터 출력 등)라면, "DB에 다시 쓰는 것"과는 다른 종류의 중복 문제다** — 영수증을 두 번 출력하는 것 자체가 문제(용지 낭비, 고객 혼란)이지 데이터 손상은 아니다. **전략 제안**: Receipt 생성에 사용된 입력(Session/Ledger 확정 데이터)이 멱등하므로(같은 입력 → 같은 영수증), "다시 생성"은 안전하나 "다시 출력(물리적 프린터 호출)"은 별도로 막아야 한다 — 예를 들어 "이 세션에 대해 영수증을 이미 발급했는가"를 별도 플래그/로그로 추적.

### Sync 중복 수행

**Sync는 외부 시스템과의 통신이라, 멱등 키(idempotency key)가 필요한 전형적인 사례다.** **전략 제안**: Session의 고유 식별자(`sessionNo` 또는 `id`)를 멱등 키로 사용해, 같은 키로 다시 보내도 외부 시스템이 중복으로 처리하지 않도록 한다 — 이는 외부 시스템과의 계약(API 설계)이 필요한 사안이라 본 앱 내부 설계만으로 끝나지 않는다.

**세 가지 모두에 공통되는 결론**: PART2의 트랜잭션 경계를 좁게(독립적으로) 두는 구조일수록 멱등성 가드를 더 많이/정교하게 만들어야 하고, 넓게(①/②처럼) 묶을수록 "부분 실패로 인한 재시도"라는 멱등성 문제의 발생 빈도 자체가 줄어든다 — **PART2와 PART5는 같은 트레이드오프의 양면**이다.

## ADR 작성 여부

**작성하지 않는다.** PART2의 권장안(②, 사실상 ①과 동일한 경계)은 "1차 권장"으로 제시됐을 뿐, **이번 단계의 목적 자체가 "확정하지 않는다"**(지시문 PART2/PART6 양쪽에서 명시)이므로 ADR을 작성할 조건("하나의 구조로 결론이 난 경우")을 충족하지 않는다. `docs/adr/ADR-007-transaction-boundary.md`는 작성하지 않으며, A-13에서 실제로 경계가 확정되는 시점에 작성한다.
