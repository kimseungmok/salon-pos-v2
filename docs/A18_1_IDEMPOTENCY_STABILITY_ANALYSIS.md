# A-18.1: Idempotency Stability Analysis

> **목적**: `SessionClosingWorkflow` 기반 구조에서 `closeSession()`이 중복 호출되더라도 데이터가 중복 생성되지 않는지 확인한다. 새 설계/계층 없음, Business Logic 변경 없음, 구현 없음(분석만).
> **대상 코드**: `lib/features/session/data/session_repository.dart`(260~298행), `lib/features/session/workflow/session_closing_workflow.dart`
> **근거**: A-14~A-17(기준 구조, 변경하지 않음), ADR-007, `docs/A13_CONCURRENCY_VALIDATION.md`(A-12.7, 동시성 분석의 직접 전신)
> 작성일: 2026-06-26

---

## PART 0 — 기준 원칙

A-14~A-17에서 확정된 구조는 변경하지 않는다. 모든 판단은 실제 코드 기준으로만 수행한다. **본 분석은 "순차 재시도"(이전 호출이 끝난 뒤 다시 호출)와 "진짜 동시 호출"(두 호출이 겹쳐서 실행)을 구분한다** — 이 둘은 서로 다른 결론을 가지며, 섞어서 판단하면 안 된다(A-12.6/A-12.7에서 이미 이 구분이 핵심이었던 것과 동일).

## PART 1 — `closeSession()` 중복 실행 경로 확인

| 항목 | 상태 | 근거 |
|---|---|---|
| Transaction 재진입 여부 | **재진입 없음** | 매 호출마다 `SessionClosingWorkflow.run()`이 별도로 `_db.transaction()`을 새로 시작한다(39행) — 같은 트랜잭션을 여러 호출이 공유하거나 재진입하는 구조가 아니다. |
| Workflow 중복 실행 여부 | **순차 재시도: 발생 안 함(가드가 막음) / 진짜 동시 호출: 발생 가능** | `closeSession()`의 가드(273~276행, `_requireSession()` 조회 후 `session.status != open` 체크)는 **`SessionClosingWorkflow` 호출(283~288행) 전에, 트랜잭션 밖에서** 수행된다. 첫 호출이 완전히 끝난 뒤(상태='closed' 커밋 완료) 재호출하면 이 가드가 즉시 막는다(순차 재시도는 안전). 그러나 두 호출이 **겹쳐서** 실행되면(Dart 단일 isolate의 협력적 동시성 — `await` 지점에서 인터리빙 가능, A-12.7에서 이미 분석됨), 둘 다 가드를 통과한 뒤 각자 `SessionClosingWorkflow.run()`을 실행할 수 있다 — **이 경로는 A-12.7 분석 이후 코드가 바뀌지 않았으므로 그대로 남아 있다.** |
| Repository 중복 write 가능성 | **순차 재시도: 없음 / 진짜 동시 호출: 있음** | 위와 동일한 이유 — 가드를 통과한 두 호출이 각자의 트랜잭션 안에서 결제수단/Ledger insert와 상태 갱신을 모두 수행하면, 두 트랜잭션 모두 정상적으로 커밋된다(DB 제약이 막지 않음, PART3). |

**핵심 발견**: A-13(Transaction 적용)과 A-14(Workflow 추출)는 **A-12.6이 식별한 "부분 실패 후 순차 재시도" 문제를 해결했다** — 가드를 통과한 뒤 쓰기 중간에 실패해도, 이제는 그 쓰기 전체가 한 트랜잭션이라 부분 커밋이 없고, 재호출 시 다시 처음부터 안전하게 실행된다. **그러나 A-12.7이 식별한 "진짜 동시 호출" 문제는 해결되지 않았다** — 가드 자체가 트랜잭션 밖에 있다는 사실은 A-13/A-14 어느 단계에서도 바뀌지 않았기 때문이다.

## PART 2 — `SessionClosingWorkflow` 중복 실행 영향

> 아래는 "진짜 동시 호출"로 두 실행이 모두 가드를 통과했다고 가정할 때의 영향이다(PART1에서 확인된 유일한 중복 발생 경로).

| 항목 | 영향 여부 | 이유 |
|---|---|---|
| Settlement 중복 생성 여부 | **영향 있음** | 두 호출 각각의 트랜잭션이 `payment_method_breakdowns`에 결제수단 행을 insert한다 — 같은 세션에 결제수단 기록이 2배로 쌓인다. |
| Ledger 중복 insert 여부 | **영향 있음**(세션에 `staff_fee` 품목이 있는 경우) | 두 호출 각각이 `_staffEarningEngine.calcEarnings()`를 호출하고 그 결과를 insert한다 — 같은 `sessionItemId`를 가진 Ledger 행이 2배로 쌓인다. |
| `StaffEarningEngine` 결과 중복 여부 | **계산 자체는 중복이 아니나, 저장이 중복됨** | `StaffEarningEngine`은 순수 함수이므로 같은 입력에 항상 같은 결과를 반환한다(계산 로직 자체에 문제는 없음) — 문제는 그 "같은 결과"가 두 번 저장된다는 점이다. |

## PART 3 — DB 레벨 중복 방지 여부 확인

| 항목 | 존재 여부 | 설명 |
|---|---|---|
| unique constraint 존재 여부 | **없음** | `lib/db/app_database.dart`/`session_tables.dart`를 확인한 결과, `payment_method_breakdowns`/`staff_earning_ledgers` 어느 테이블에도 중복 삽입을 막는 UNIQUE 제약이 없다(A-12.9/A-13.6에서 이미 "DB Constraint는 추가되지 않았다"로 확인된 사실의 재확인 — A-14~A-17 사이에도 추가되지 않았다). |
| transaction rollback 보호 범위 | **한 번의 호출 내부에서만** | `_db.transaction()`은 **그 콜백 하나의 원자성**만 보장한다 — "이 세션은 이미 처리 중이니 다른 호출은 거부한다"는 식의 **호출 간(cross-call) 배타성**은 트랜잭션 메커니즘 자체가 제공하지 않는다. |
| session 상태 기반 방어 여부 | **존재하나 범위가 트랜잭션 밖** | `session.status != open` 가드(273~276행)가 유일한 방어 장치이며, PART1에서 확인한 대로 이 가드는 트랜잭션 시작 전에 읍어서 판단하는 "확인 후 사용"(TOCTOU) 패턴이다. |

**종합**: 현재 구조에서 중복을 막는 장치는 **오직 `session.status` 가드 하나**이며, 그 가드가 트랜잭션과 결합돼 있지 않다는 것이 A-12.7부터 지금까지 변하지 않은 유일한 약점이다.

## PART 4 — Workflow 재실행 안전성 확인

| 항목 | 결과 | 근거 |
|---|---|---|
| partial execution 발생 여부 | **발생 안 함(각 호출 단위로는)** | 한 번의 `SessionClosingWorkflow.run()` 호출은 항상 전부 성공 또는 전부 실패한다(트랜잭션 원자성, A-17에서 이미 확인) — "재실행"이라는 행위 자체가 깨진 중간 상태를 만들지는 않는다. |
| side-effect 중복 여부 | **두 호출이 겹칠 때만 발생** | PART2에서 확인한 그대로 — 단, 이는 "재실행이 위험하다"는 뜻이 아니라 "재실행을 막는 장치가 트랜잭션 밖에 있어서, 막혀야 할 재실행이 안 막히는 경우가 있다"는 뜻이다. |
| 예외 발생 시 상태 일관성 | **개별 호출 단위로는 일관적, 누적 관점에서는 별개 문제** | 한 호출이 실패하면 그 호출의 모든 쓰기가 rollback되어 DB는 항상 "이전 상태" 또는 "완전히 새 상태" 중 하나로 깨끗하게 남는다(일관성 유지). 다만 "여러 호출에 걸친 데이터 중복"은 각 호출이 개별적으로는 모두 "정상 성공"이었기 때문에 발생하는 문제라, 예외/rollback 메커니즘이 다룰 수 있는 범위 밖이다. |

## PART 5 — Idempotency 위험 판단

| 항목 | 위험 여부 | 이유 |
|---|---|---|
| 중복 실행 시 데이터 불일치 가능성 | **있음(진짜 동시 호출 시에만)** | PART1/2에서 확인한 대로 가드가 트랜잭션 밖에 있어 TOCTOU 레이스가 그대로 남아 있다 — A-12.7 시점과 코드 수준에서 동일하다. |
| rollback으로도 복구 불가능한 상태 존재 여부 | **없음** | 중복이 발생하는 시나리오는 "두 트랜잭션이 각각 정상적으로 성공"한 경우이지, "깨진 채로 남는" 경우가 아니다 — rollback이 필요한 상황 자체가 아니므로 "복구 불가능"이라는 표현이 적용되지 않는다. 문제는 복구가 아니라 **예방**의 영역이다. |
| 운영 환경에서 재시도 안전성 | **조건부** | **순차 재시도(이전 호출이 완전히 끝난 뒤 재시도)는 안전하다** — 가드가 정상 작동한다. **동시 재시도(겹친 호출)는 안전하지 않다** — 단, `docs/A13_IMPACT_MAPPING.md`(A-12.9)에서 이미 확인된 사실대로 **현재 `closeSession()`을 호출하는 production 화면이 0건**이므로, 이 위험의 실제 노출도는 지금 0이다. 이는 ADR-007이 이미 "현재 호출자가 없어 동시성 노출도가 0"이라는 근거로 동시성 대응을 A-14 이후로 명시적으로 이관한 것과 정확히 같은 사안이다 — 본 문서가 새로 발견한 위험이 아니라, **A-12.7/ADR-007이 이미 알고 있던 위험이 A-14~A-17 사이에도 해결되지 않은 채 그대로 남아 있다는 사실의 재확인**이다. |

## PART 7 — 최종 결론

| 확인 항목 | 결과 |
|---|---|
| `closeSession()`은 idempotent한가 | **부분적으로만 그렇다.** 순차 재시도(이전 호출 완료 후 재호출)에는 idempotent하다(가드가 정상 작동, A-13의 Transaction 적용으로 부분 실패 후 재시도 문제도 해소됨). **진짜 동시 호출에는 idempotent하지 않다** — 가드가 트랜잭션 밖에 있어 TOCTOU 레이스가 그대로 존재한다(PART1/5). |
| Workflow 재실행은 안전한가 | 개별 호출 단위로는 안전하다(원자성 보장, PART4) — 그러나 "두 호출이 동시에 안전하게 공존하는가"라는 의미로는 안전하지 않다. |
| Transaction이 중복 실행을 방지하는가 | **방지하지 않는다.** Transaction은 "한 호출 내부의 원자성"만 보장하며, "호출 간 배타성"(누가 먼저 처리할 권리를 갖는지)은 보장하지 않는다(PART3). 이는 A-12.7에서 이미 도출된 결론과 동일하며, A-13/A-14 적용 이후에도 바뀌지 않았다. |
| 운영 환경에서 재호출이 허용 가능한가 | **순차 재호출은 허용 가능, 동시 재호출은 허용 불가** — 단 현재 호출자가 없어(A-12.9) 이 위험이 실제로 발생할 경로가 지금은 없다(PART5). |

### 결론

4개 확인 항목 중 마지막 항목을 제외한 나머지가 **조건부**(순차 vs 동시 구분 필요)로 나타났고, "Transaction이 중복 실행을 방지하는가"는 명확히 **아니오**로 확인되었다. 따라서 본 문서는 **"Idempotency Stability Confirmed"를 명시하지 않는다.**

**정확한 결론은 다음과 같다**: 현재 구조는 **"순차 재시도에 대한 Idempotency는 확보됐으나, 동시 호출에 대한 Idempotency는 확보되지 않았다"**(Partial). 이는 A-12.7/ADR-007이 이미 식별하고 "현재 호출자가 없다"는 근거로 A-14 이후로 명시적으로 이관한 동일한 위험이며, A-14~A-17(Workflow 추출/패턴 검증/아키텍처 확정/운영 안정성 확인)의 어느 단계에서도 이 위험을 해소하는 변경은 없었다 — **새로운 결함이 아니라, 알려진 채로 의도적으로 남겨진 미해결 항목의 재확인이다.**
