# A-12.7: Concurrency & Idempotency Validation(A-13 착수 전)

> **목적**: `closeSession()`이 동시 실행(Concurrency) 환경에서도 안전한지 실제 코드 기준으로 검증한다. 새 기능 설계 없음, 코드/DB/테스트 변경 없음.
> **대상 코드**: `lib/features/session/data/session_repository.dart` 247~327행(`closeSession()`)
> **선행 문서**: `docs/A13_TRANSACTION_BOUNDARY_REVIEW.md`(A-12.5), `docs/A13_FINANCIAL_WORKFLOW_STATE_ANALYSIS.md`(A-12.6 — 9단계 분해, 위험 구간 5~8번 식별)
> 작성일: 2026-06-26

---

## 실행 환경 전제(분석의 기반)

이 앱은 **Flutter/Dart 단일 isolate**에서 동작하는 오프라인 우선 SQLite 단일 인스턴스다. 이는 "동시성"의 의미를 일반적인 멀티스레드/멀티프로세스 서버와 다르게 만든다:

- Dart의 단일 isolate는 **협력적 동시성(cooperative concurrency)**이다 — 진짜 OS 스레드가 동시에 메모리를 건드리는 일은 없다. 대신 `async`/`await`로 표시된 지점에서만 다른 비동기 작업에 실행을 양보(yield)한다. **`await`가 없는 구간은 원자적으로(끊김 없이) 실행된다.**
- 따라서 "Thread A / Thread B 동시 실행"은 이 앱에서는 "두 개의 `closeSession()` 호출이 서로의 `await` 지점에서 번갈아 진행되는 것"을 의미한다 — 진짜 동시 메모리 접근이 아니라 **인터리빙(interleaving)**이다.
- SQLite 자체는 단일 쓰기자(single-writer) 모델이라 디스크에 쓰는 시점은 직렬화되지만, **애플리케이션 로직(읍기→판단→쓰기)이 직렬화되는 것은 아니다** — 이 차이가 본 문서의 핵심이다.

## PART 1 — Concurrency 시나리오 분석

### 1. Thread A / Thread B 동시 실행(같은 세션에 `closeSession()` 두 번 호출)

**현재 코드는 안전하지 않다.** A-12.6에서 식별한 "위험 구간"(5~8번 단계, `await`를 포함한 여러 단계)이 정확히 인터리빙이 일어날 수 있는 지점이다:

1. 호출 A가 260~263행(상태 가드)에서 `session.status == 'open'`을 확인하고 통과한다.
2. 호출 A가 271행(`_db.batch()`, `await`)에서 이벤트 루프에 제어를 양보한다.
3. 이 틈에 호출 B가 시작되어, 260~263행에서 **같은 세션을 다시 조회** — 이때 호출 A는 아직 315행(상태 변경)에 도달하지 못했으므로 DB의 `status`는 여전히 `'open'`이다. **호출 B도 가드를 통과한다.**
4. 이제 A와 B 둘 다 "마감 가능"으로 판단된 채로 각자 271~320행(결제수단 insert, Ledger 계산/insert, 상태 변경)을 **각자 끝까지** 실행한다.

**결과**: `payment_method_breakdowns`에 결제수단 기록이 2배, (해당 세션에 `staff_fee` 품목이 있다면) `staff_earning_ledgers`에 수당 기록이 2배로 쌓인다. 상태 변경(315~320행)은 양쪽 다 같은 값(`'closed'`)을 쓰므로 그 자체로는 깨지지 않지만, **이미 그 전에 금전 데이터가 중복 기록된 뒤다.**

### 2. UI 중복 클릭

**시나리오 1과 동일한 메커니즘**이며, 이 앱(단일 사용자 POS, 멀티 디바이스/서버 없음)에서 **가장 현실적으로 발생 가능한 경우**다 — 사용자가 "마감" 버튼을 두 번 빠르게 누르면, 화면이 버튼을 비활성화하기 전(또는 비활성화 로직이 없다면) 두 번의 `closeSession()` 호출이 거의 동시에 시작되어 시나리오 1과 같은 인터리빙이 일어난다. **현재 `SessionRepository` 코드만으로는 막을 수 없다** — 이 가드는 UI 계층(버튼 비활성화)에 의존하고 있을 가능성이 높고, `SessionRepository` 자체에는 그런 방어가 없다.

### 3. 네트워크 재전송

**현재 이 앱에는 해당 사항이 없다.** 오프라인 우선 단일 SQLite 인스턴스로, 클라이언트-서버 동기화 계층 자체가 아직 존재하지 않는다(A-12.5/A-12.6에서 반복 확인된 그대로). 향후 Sync 기능이 추가되어 "서버에 보냈는데 응답을 못 받아서 재전송"하는 흐름이 생기면, 그 재전송이 `closeSession()`을 다시 호출하는 형태일지 아니면 이미 닫힌 세션의 동기화만 재시도하는 형태일지에 따라 적용 가능성이 달라진다 — **지금은 분석 대상 자체가 없다.**

### 4. 앱 재실행 후 재시도

**시나리오 1/2와는 다른 종류다 — 동시 실행이 아니라 순차적 재시도다.** A-12.6 PART2에서 이미 분석된 경우(5~8번 단계 중 실패 후 앱이 죽고, 재실행 후 사용자가 다시 마감을 시도)와 정확히 같다. **차이점**: 시나리오 1/2는 "둘 다 살아서 인터리빙"하는 것이고, 시나리오 4는 "하나가 죽은 뒤 새로 하나가 시작"하는 것이다 — 그러나 **현재 코드 기준으로 둘 다 같은 결과(중복 insert)를 만든다.** 코드 입장에서는 "이전 호출이 살아있는 채로 겹쳤는지" 또는 "이전 호출이 죽고 새 호출이 시작됐는지"를 구분할 방법이 전혀 없기 때문이다 — 두 경우 모두 "세션 상태가 여전히 `open`인데 일부 데이터는 이미 들어가 있는 상태"에서 재호출이 일어난다는 점이 동일하다.

## PART 2 — Guard 계층 비교

| 계층 | 막을 수 있는 것 | 막지 못하는 것 |
|---|---|---|
| **Application Guard**(예: "이 세션에 이미 Ledger가 있는가" 사전 조회 후 분기) | 이미 완전히 끝난 호출 뒤의 **순차적** 재호출(예: 시나리오 4에서, 이전 호출이 5~8번을 전부 끝낸 뒤 재호출되는 경우) | **시나리오 1/2(진짜 동시 실행)** — Guard의 "조회"도 비동기 작업이라, A와 B가 둘 다 "아직 Ledger 없음"을 확인한 뒤에야 각자 insert를 시작할 수 있다(Guard 자체가 같은 종류의 TOCTOU 레이스에 노출됨). 즉 Guard만으로는 **이 문서의 핵심 위험을 막지 못한다.** |
| **Transaction**(4~8번 단계를 하나의 DB 트랜잭션으로 묶음, A-12.5/A-12.6에서 1차 권장) | 한 호출 내부의 **부분 실패**(A-12.6의 핵심 문제 — 결제수단만 저장되고 Ledger/상태변경이 안 되는 상태) | **동시 실행 자체는 막지 못한다** — 트랜잭션은 "한 호출의 쓰기들이 원자적으로 전부 성공/실패"하게 만들 뿐, "두 호출이 동시에 각자의 트랜잭션을 시작해서 둘 다 성공"하는 것은 그대로 허용한다(트랜잭션 안에서 상태를 **다시 검증**하지 않는 한). |
| **DB Constraint**(예: 특정 컬럼 조합에 UNIQUE) | **진짜 동시 실행에도 안전** — SQLite가 insert 시점에 제약을 검사하므로, A와 B가 동시에 같은 조건으로 insert를 시도해도 둘 중 하나는 반드시 제약 위반으로 실패한다(애플리케이션 로직의 레이스 윗도우와 무관하게 DB 레벨에서 막힘). 본 비교에서 **유일하게 동시성 자체에 면역인 계층.** | `payment_method_breakdowns`처럼 "정상적으로 한 세션에 여러 행이 있을 수 있는"(분할결제) 테이블에는 단순 UNIQUE 설계가 부적합하다(A-12.6에서 이미 식별) — 제약의 키를 무엇으로 잡을지 별도 설계가 필요하다. 또한 실패한 쪽 호출에 대한 에러 처리(사용자에게 무엇을 보여줄지)가 별도로 필요하다. |
| **Idempotency Key**(호출자가 같은 "시도"임을 식별할 키를 전달, 시스템이 "이미 처리됨"을 판정) | 호출자가 재시도/중복 클릭을 **같은 키**로 표시할 수 있다면, 그 키 자체에 DB Constraint를 걸어 동시 실행에도 안전하게 막을 수 있다(사실상 DB Constraint를 "키 컬럼"에 적용하는 특수 사례). | 키가 없으면 무용하다 — **이 세션을 다시 닫으려는 시도는 항상 같은 키(예: `sessionId` 자체)를 의미한다고 본다면, `sessionId`를 키로 쓰는 순간 DB Constraint 계층으로 귀결된다.** 즉 이 앱의 `closeSession()` 맥락에서는 Idempotency Key가 별도의 새 메커니즘이 아니라, **DB Constraint를 적용할 "키"를 무엇으로 잡을지에 대한 관점**에 가깝다. |

### 표가 보여주는 결론

**Application Guard와 Transaction은 둘 다 "동시 실행" 문제 자체를 해결하지 못한다** — Guard는 같은 종류의 레이스에 노출되고, Transaction은 한 호출 내부의 원자성만 보장한다. **DB Constraint(또는 그 적용 형태인 Idempotency Key)만이 진짜 동시 실행에도 면역이다.**

## PART 3 — Race Condition 분석(A-12.6 단계 분해를 동시성 관점에서 재확인)

A-12.6 PART1의 9단계 표를 그대로 가져와, "동시성 관점"에서 각 `await` 지점이 인터리빙 윗도우(window)가 되는지 표시한다.

| # | 단계 | `await` 포함 | 동시성 윗도우인가 |
|---|---|---|---|
| 1 | 입력 검증 | 아니오 | 아니오(DB 미접근, 순수 동기 로직) |
| 2 | 세션 조회 + 상태 가드 | **예**(`await _requireSession()`) | **예 — 이게 레이스의 시작점.** 이 `await`에서 다른 호출이 끼어들 수 있고, 그 다른 호출도 똑같이 이 지점에서 `'open'`을 읍게 된다. |
| 3 | Settlement 계산 | 아니오 | 아니오(메모리 내 계산) |
| 4 | 결제수단 insert | **예** | 예 — 단, 이 지점에서의 인터리빙 자체가 문제를 일으키는 건 아니다(이미 2번에서 양쪽 다 가드를 통과한 뒤이므로, 이 시점의 인터리빙은 "이미 발생한 레이스의 결과"를 두 번 쓰는 것일 뿐). |
| 5 | 품목 조회 | 예 | 예(4번과 동일한 이유) |
| 6 | Engine 계산 | 아니오 | 아니오(순수 계산) |
| 7 | Ledger insert | 예 | 예(4번과 동일한 이유) |
| 8 | 상태 변경 | 예 | 예 — **이 지점도 한 번 더 살펴볼 가치가 있다.** A의 8번 쓰기와 B의 8번 쓰기가 둘 다 일어나면, 최종적으로 `status`/`endAt`/`updatedAt`은 "나중에 실행된 쪽"의 값으로 남는다(SQLite가 두 `UPDATE`를 직렬화하므로 데이터 자체가 깨지진 않지만, 두 호출 모두 "성공"으로 보일 수 있다는 점이 문제). |
| 9 | 재조회 | 예 | 아니오(이미 끝난 데이터를 읍는 것뿐, 레이스에 영향 없음) |

### 결론 — 레이스의 "근본 원인"은 단 하나

**2번(상태 확인)과 8번(상태 쓰기) 사이에, 그 둘을 하나로 묶는 원자적 연산이 없다는 것**이 유일한 근본 원인이다. 이는 일반적으로 **TOCTOU(Time-Of-Check to Time-Of-Use)** 레이스라고 부르는 패턴과 정확히 일치한다 — "확인"과 "사용(반영)" 사이에 시간차가 있고, 그 사이에 다른 실행이 끼어들 수 있다. A-12.6이 "실패로 인한 순차 재시도"의 위험 구간으로 짚은 5~8번과, 본 PART가 "동시 실행"의 위험 구간으로 짚은 2~8번은 **같은 코드 구간을 다른 트리거(실패 후 재시도 vs 진짜 동시 실행)로 바라본 것**이며, 둘 다 결국 "2번과 8번이 원자적으로 묶여 있지 않다"는 같은 원인에서 나온다.

## PART 4 — A-13 최소 구현 범위(분석만, 구현하지 않음)

### Transaction만으로 충분한가? — **아니오**

PART2/PART3에서 확인한 그대로, Transaction은 "부분 실패"(A-12.6의 문제)를 해결하지만 "동시 실행"(본 문서의 문제)은 해결하지 못한다. **A-12.5/A-12.6의 1차 권장(4~8번을 하나의 Transaction으로)이 틀린 것은 아니다 — 다만 그 권장은 실패-후-재시도 문제를 향한 것이었고, 동시 실행 문제에는 답이 되지 못한다는 점을 본 문서가 보완한다.**

### DB Constraint도 필요한가? — **이 문서의 분석상 그렇다(다만 구체적 설계는 A-13에서)**

PART2 표가 보여주듯, 진짜 동시 실행에 면역인 계층은 DB Constraint(또는 그 적용 형태인 Idempotency Key)뿐이다. 가장 자연스러운 후보는 **"상태 확인과 상태 변경을 하나의 원자적 SQL 연산으로 합치는 것"**이다 — 예를 들어 "`UPDATE payment_sessions SET status='closed' WHERE id=? AND status='open'`을 실행하고, 영향받은 행 수가 0이면 '이미 다른 호출이 마감했다'고 판단해 중단한다"는 조건부 업데이트(optimistic locking과 유사한 패턴). 이는 SQLite의 **단일 `UPDATE` 문 자체가 원자적**이라는 보장을 이용하는 것으로, 별도의 새 컬럼/제약을 추가하지 않고도 2~8번의 "확인-사용" 간극을 없앨 수 있는 방향이다. **단, 이 방식을 택하면 현재 코드의 실행 순서(상태 변경이 8번, 워크플로의 끝)를 "상태 변경을 먼저 시도하고 통과해야 나머지를 진행"하는 순서로 재배치해야 하므로, 이는 단순한 "보호 장치 추가"를 넘어서는 흐름 변경이다 — A-13에서 신중히 검토할 사안으로 남긴다(본 문서는 결론을 내리지 않는다).**

### Application Guard를 유지해야 하는가? — **단독으로는 불충분, 보조적 역할은 가능**

PART2에서 확인했듯 Guard 자체는 동시 실행을 막지 못한다. 다만 "명백히 이미 닫힌 세션"(예: 시나리오 4처럼 완전히 끝난 뒤의 순차 재호출)에 대해서는 현재의 261~263행 가드가 이미 그 역할을 하고 있다 — **이 가드를 없앨 이유는 없지만, 이것만으로 동시 실행 문제가 해결된다고 간주해서는 안 된다.**

### 종합(결론 — 분석 의견, 확정 아님)

이 세 가지 질문에 대한 본 문서의 분석을 종합하면: **Transaction(부분 실패 방지)과 DB Constraint류의 원자적 조건부 갱신(동시 실행 방지)은 서로 다른 문제를 풀기 위한 별개의 장치이며, 둘 중 하나만으로는 A-13이 직면한 두 문제(A-12.6의 부분 실패 + A-12.7의 동시 실행)를 모두 풀 수 없다.** A-13은 이 둘을 함께 검토해야 한다 — 다만 구체적 구현(어디에 트랜잭션을 두고, 어떤 SQL로 조건부 갱신을 할지)은 본 문서의 범위 밖이다.
