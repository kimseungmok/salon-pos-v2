### 복합 성격을 가진 단계

- **단계 2**: "DB 영속화"(조회)이면서 동시에 그 결과로 "상태 판정"(이후 단계를 진행할지 결정)을 한다 — 쓰기는 아니지만 워크플로 전체의 분기점이라는 점에서 단순 조회와 다른 무게를 가진다.
- **단계 8**: "DB 영속화"(쓰기)이면서 그 자체가 "상태 변경"이다 — 워크플로 전체에서 유일하게 `payment_sessions.status` 값을 바꾸는 지점이며, **이 워크플로의 마지막 쓰기 단계**다.

### 핵심 관찰 — 상태 변경(8번)이 워크플로의 끝에 있다

순수 계산(1, 3, 6) → DB 영속화/쓰기(4, 7) → **상태 변경(8)** 순서다. 즉 **"마감되었다"는 사실(상태='closed')은 결제 기록(4)과 수당 기록(7)이 모두 끝난 뒤에야 확정된다.** 이는 A-12.5에서 이미 식별된 순서이며, 본 PART에서 라인 단위로 재확인했다.

## PART 2 — 단계별 실패 분석(실제 코드 기준)

> 가정: 표의 "실패 단계"에서 예외가 발생해 `closeSession()`이 그 지점에서 중단된다(코드의 `try`가 `AppException`은 `rethrow`, 그 외는 `DatabaseException.writeFailed`로 포장하지만 — **어느 쪽이든 함수는 그 시점에서 중단되고 이후 단계는 실행되지 않는다.** `_db.batch()`를 감싸는 트랜잭션도 없으므로, 이미 끝난 단계의 커밋을 되돌리는 메커니즘 자체가 없다).

| 실패 단계 | 이미 저장된 데이터 | 아직 저장 안 된 데이터 | Session 상태 | 재호출 가능한가 | 중복 Ledger 위험 |
|---|---|---|---|---|---|
| 1(입력 검증) | 없음 | 전부 | `open`(불변) | 가능 — 입력만 고치면 재호출 가능 | 없음(아무것도 안 됨) |
| 2(세션 조회/가드) | 없음 | 전부 | `open`(불변, 단 `BusinessRuleException`이면 이미 `closed`/`cancelled`라는 뜻) | 세션이 이미 `closed`/`cancelled`라면 가드가 막아서 의미 없는 재호출 | 없음 |
| 3(Settlement 검증) | 없음 | 전부 | `open`(불변) | 가능 — 결제수단 금액을 고쳐서 재호출 가능 | 없음 |
| 4(결제수단 insert) | **없음**(이 단계 자체가 실패) | `payment_method_breakdowns`, `staff_earning_ledgers`, 상태 변경 전부 | `open`(불변) | 가능 | 없음 |
| 5(품목 조회) | `payment_method_breakdowns`(4에서 이미 커밋됨) | `staff_earning_ledgers`, 상태 변경 | `open`(불변) | **가능 — 그리고 이게 문제다.** 재호출 시 4번이 다시 실행되어 **`payment_method_breakdowns`에 중복 행이 또 insert된다**(결제수단 기록 중복). | 아직 없음(이 단계는 Ledger 이전) |
| 6(Engine 계산) | `payment_method_breakdowns` | `staff_earning_ledgers`, 상태 변경 | `open`(불변) | 가능(위와 동일한 문제) | 아직 없음 |
| 7(Ledger insert) | **`payment_method_breakdowns`** | 상태 변경만 | `open`(불변) | **가능 — 가장 위험한 지점.** 재호출 시 4번(결제수단 중복)과 7번(Ledger 중복)이 **둘 다** 다시 실행된다. | **있음.** 첫 시도에서 7번까지는 성공했는데(Ledger insert 완료) 그 직후(예: 8번 직전) 실패했다면, 세션은 여전히 `open`이라 재호출이 가능하고, 재호출은 5~7번을 처음부터 다시 실행해 **같은 세션에 Ledger가 2배로 쌓인다.** |
| 8(상태 변경) | `payment_method_breakdowns`, `staff_earning_ledgers` | 없음(상태 변경만 실패) | **`open`으로 남음**(`closed`로 바뀌기 전에 실패) | **가능 — 7번과 동일한 중복 위험.** | **있음**(7번과 동일) |
| 9(재조회) | 전부 저장됨(`status`도 이미 `closed`로 커밋됨) | 없음 | **`closed`**(이미 바뀜) | **불가능 — 가드(단계 2)가 `BusinessRuleException`을 던져 막는다.** | **없음** — 이 시점부터는 안전하다(상태가 이미 `closed`이므로 재호출 자체가 막힘. 단계 9 자체의 실패는 데이터 정합성에는 영향 없고, 호출자가 최신 상태를 못 돌려받는 것뿐). |

### 종합 — "위험 구간"은 정확히 5·6·7·8번이다

**상태가 `closed`로 바뀌기 전(8번 완료 전)까지는 재호출이 항상 가능하고, 그 재호출은 이미 커밋된 4번(결제수단)과 7번(Ledger, 조건이 맞으면)을 처음부터 다시 실행한다.** 이것이 A-12.5에서 이미 지적한 위험을 단계 번호로 정확히 좁힌 것이다 — **"5번 이후, 8번 이전"의 모든 실패 지점**에서 재호출 시 중복이 발생한다.

## PART 3 — State Machine 분석(결론 내리지 않음)

### ① `OPEN → CLOSED`(현재 구조)

| 기준 | 내용 |
|---|---|
| 장점 | 가장 단순 — 상태 값이 2개뿐(`open`/`closed`, `cancelled` 포함 3개), 기존 코드/테스트가 전부 이 모델을 전제로 작성됨. |
| 단점 | **PART2에서 드러난 정확한 문제** — "마감 절차가 진행 중"이라는 상태가 없어서, 절차 도중 실패해도 외부에서 보기엔 여전히 `open`이다. 즉 "아직 마감 시도 전"과 "마감 시도했다가 중간에 실패함"을 상태값으로 구분할 수 없다. |
| 구현 영향 | 없음(이미 구현됨, 변경 시에는 기존 21개+ 테스트의 상태 가정을 재검토해야 함). |
| A-13 영향 | PART2의 중복 위험을 상태 모델만으로는 막을 수 없다 — 별도의 Idempotency 장치(PART4)가 필수가 된다. |

### ② `OPEN → CLOSING → CLOSED`

| 기준 | 내용 |
|---|---|
| 장점 | "마감 절차가 시작됐다"는 사실 자체를 영속화할 수 있다 — `CLOSING` 상태에서 실패하면, 재호출 시 "이미 `CLOSING`이었으니 처음부터 다시 하지 말고 이어서 하거나 거부한다"는 판단이 가능해진다. |
| 단점 | `CLOSING`에서 멈춘 세션을 어떻게 처리할지(이어서 진행? 되돌려서 `OPEN`으로? 운영자 개입 필요?)라는 **새로운 정책 질문**이 생긴다 — 상태를 추가하는 것 자체가 또 다른 미결 사항을 만든다. 기존 `SessionStatus` enum(`open`/`closed`/`cancelled`)과 모든 참조 코드(`addItem()`/`cancelSession()`의 가드 등)에 새 상태를 끼워 넣어야 함 — 영향 범위가 큼. |
| 구현 영향 | 크다 — enum 추가, 모든 상태 분기 재검토, 마이그레이션(기존 `open`/`closed`/`cancelled` 데이터는 그대로 두되 새 상태가 끼어들 자리를 코드 전체에서 인지해야 함). |
| A-13 영향 | 중복 방지는 쉬워지지만, "`CLOSING`에서 멈춘 세션의 운영 정책"이라는 새로운 미결 사항이 A-13에 추가된다. |

### ③ 현재 상태 유지 + Workflow 진행 여부를 별도 관리

| 기준 | 내용 |
|---|---|
| 장점 | `SessionStatus`(사업적 의미: 영업 관점에서 열려있다/닫혔다)와 "이 마감 절차가 어디까지 진행됐는가"(기술적 의미: 워크플로 진행 상태)를 **분리**할 수 있다 — ②처럼 사업 상태 enum을 건드리지 않아도 됨. 예를 들어 별도의 진행 마커(예: "Ledger까지 완료" 같은 체크포인트)를 두면, 재호출 시 그 마커를 보고 "이미 한 일은 건너뛰고 안 한 일만 한다"는 판단이 가능. |
| 단점 | 그 "별도 관리"가 정확히 무엇인지(새 컬럼? 별도 테이블? 메모리상 추적?)가 정해지지 않으면 막연한 방향일 뿐이다 — ②보다 구체성이 낮다. |
| 구현 영향 | 마커의 형태에 따라 다름 — 작은 컬럼 추가 정도면 영향이 작지만, 별도 테이블이면 ②에 준하는 영향이 생김. |
| A-13 영향 | "마커를 어떻게 설계할지"가 A-13의 새로운 결정 사항이 된다 — ②의 문제(끼어든 상태를 어떻게 운영할지)와 본질적으로 같은 종류의 질문을 다른 형태로 미루는 것에 가깝다. |

**비교만 하고 결론 내지 않는다** — 지시문 그대로, 본 PART는 세 후보의 트레이드오프를 나열할 뿐 A-13에서 어느 쪽을 택할지는 여기서 정하지 않는다.

## PART 4 — Idempotency 분석(권장하되 확정하지 않음)

| 항목 | 중복 실행 시 실제로 발생하는 일(PART2 기준) | 후보 방지 전략 |
|---|---|---|
| **Ledger** | 같은 세션에 `staff_earning_ledger` 행이 2배(또는 그 이상)로 쌓인다(PART2, 5~8번 구간 재실행) | (a) **Application Guard**: `closeSession()` 진입 시 "이 세션에 이미 Ledger가 있는가"를 조회해서 있으면 건너뛴다. (b) **DB UNIQUE Constraint**: 예를 들어 `(sessionId, sessionItemId)` 조합에 유니크 제약을 걸어 DB 레벨에서 중복 insert를 거부한다(실패를 에러로 만들 뿐, 우아하게 막지는 못함 — 그래도 안전망은 됨). (c) **Workflow 상태 관리**(PART3 ②/③): 애초에 재실행 자체가 안 일어나게 막는다. |
| **Settlement**(`payment_method_breakdowns`) | 같은 세션에 결제수단 기록이 중복 insert된다(PART2, 4번 이후 구간 재실행) — Ledger와 동일한 패턴의 위험 | Ledger와 동일한 후보(a/b/c) — 특히 (b)는 "같은 세션에 같은 결제수단이 두 번 기록되는 것"을 막기엔 자연스럽지 않다(분할결제는 같은 `method`가 여러 행으로 정상적으로 존재할 수 있으므로 UNIQUE 제약 설계가 까다로움) — 이 항목은 (a) 또는 (c)가 더 적합해 보인다(확정 아님). |
| **Receipt**(미구현) | 같은 영수증이 여러 번 "출력"되는 외부 부수효과(A-12.5 PART5와 동일 결론) | **Idempotency Key**: 세션의 고유 식별자(`sessionNo`)를 키로 사용해 "이미 발급된 영수증인가"를 추적 — DB 영속화가 아니라 외부 출력 호출 직전의 멱등 체크에 적합. |
| **Sync**(미구현) | 외부 시스템에 같은 데이터가 중복 전송됨(A-12.5 PART5와 동일 결론) | **Idempotency Key**: 외부 API가 멱등 키를 지원한다는 전제 하에 `sessionNo`/`id`를 키로 사용 — 이건 본 앱 내부 설계만으로 끝나지 않고 외부 시스템과의 계약이 필요(A-12.5에서 이미 언급). |

### 권장(확정 아님)

**Ledger/Settlement에는 (a) Application Guard를 1차 권장한다** — 이유: (b) DB UNIQUE Constraint는 Settlement 쪽에서 분할결제라는 정상 케이스와 충돌할 위험이 있고, (c) Workflow 상태 관리(PART3)는 아직 어느 State Machine 후보를 쓸지 정해지지 않아 전제가 없다. Application Guard는 **지금의 State Machine(①, 변경 없음)을 유지한 채로도 적용 가능한 가장 점진적인 방법**이라는 점에서 1차 후보로 제시하나, **A-13에서 PART3의 결론과 함께 재검토해야 한다**(특히 ②/③을 택하면 Guard 없이도 구조적으로 막을 수 있어, Guard가 불필요해질 수 있다).

## PART 5 — Transaction Boundary 재검토(A-12.5 기반 재분류)

A-12.5(`A13_TRANSACTION_BOUNDARY_REVIEW.md`)의 결론(① ≈ ② 수렴, Receipt/Sync는 트랜잭션 밖)을 본 PART의 9단계 분해에 그대로 매핑한다.

| 분류 | 해당 단계 | 분류 근거 |
|---|---|---|
| **반드시 같은 Transaction** | 4(결제수단 insert), 5~7(Ledger 계산+insert), 8(상태 변경) | PART2에서 확인된 그대로 — 이 구간(정확히 "4번부터 8번까지")이 부분적으로만 커밋되면 PART2의 모든 중복/불일치 위험이 발생한다. 이 4개 단계가 "전부 성공 또는 전부 실패"로 묶여야 PART2의 위험 구간(5~8번)이 원천적으로 사라진다. |
| **Retry 가능한 처리** | 1(입력 검증), 2(세션 조회/가드), 3(Settlement 계산) | DB 쓰기가 없는 순수 계산/조회 단계라, 실패해도 아무것도 변경되지 않으므로 그냥 다시 호출하면 된다 — "Retry"라는 개념이 적용될 대상조차 거의 아니다(애초에 부수효과가 없음). |
| **비동기 처리 가능한 작업** | (현재 코드에는 없음) Receipt/Sync(A-12.5에서 분류된 미구현 단계) | 핵심 트랜잭션(4~8번)이 끝난 뒤, 그 결과(확정된 Session/Ledger)를 입력으로 받아 독립적으로 수행될 수 있는 외부 연동이기 때문 — 이 작업들이 실패해도 핵심 트랜잭션을 되돌릴 이유가 없다(A-12.5 PART3과 동일 결론). |

**9(재조회)는 분류에서 제외한다** — 이미 커밋된 데이터를 다시 읽는 것뿐이라, 트랜잭션 경계 논의와 무관하다(실패해도 데이터에 영향 없음, PART2에서 이미 "정합성에는 영향 없음"으로 확인).

## PART 6 — A-13 구현 체크리스트(새로운 설계 추가 없음, 결정 항목만 나열)

| 항목 | 본 문서/선행 문서에서 이미 분석된 내용 | A-13이 결정할 것 |
|---|---|---|
| **Transaction Boundary** | PART5 — "4~8번은 반드시 같은 Transaction" | 실제로 `_db.transaction()`(또는 동등한 메커니즘)으로 묶을지, 그 구체적 구현 방법 |
| **State Machine 필요 여부** | PART3 — ①(현행)/②(`CLOSING` 추가)/③(별도 마커) 3개 후보와 트레이드오프 | 어느 후보를 택할지(또는 ①을 유지하고 PART4의 Guard만으로 충분한지) |
| **Idempotency 전략** | PART4 — Ledger/Settlement는 Application Guard 1차 권장(확정 아님), Receipt/Sync는 Idempotency Key | Guard의 구체적 조건(예: "Ledger 존재 여부"를 정확히 어떤 쿼리로 판단할지), 또는 State Machine 선택에 따라 Guard 자체가 불필요해지는지 |
| **DB Constraint 필요 여부** | PART4 — UNIQUE Constraint는 Settlement(분할결제 케이스)에는 부적합 가능성, Ledger에는 적용 가능성 있음(미확정) | Ledger 테이블에 `(sessionId, sessionItemId)` UNIQUE를 실제로 추가할지, 추가한다면 기존 데이터(이미 A-12에서 생성된 Ledger)와의 마이그레이션 영향 |

본 체크리스트는 PART1~5에서 이미 분석된 사실/트레이드오프를 다시 나열한 것이며, 새로운 설계 항목은 추가하지 않았다.
