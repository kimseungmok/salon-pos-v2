> **목적**: A-12.5~A-12.9에서 검토된 내용 중, A-13 구현에 **반드시 필요한 결정만** 확정한다. 새 기능/아키텍처/패턴 제안 없음. 코드/DB/테스트/ADR 수정 없음(신규 ADR 작성은 별도 판단).
> **근거 문서**: `A13_TRANSACTION_BOUNDARY_REVIEW.md`(A-12.5), `A13_FINANCIAL_WORKFLOW_STATE_ANALYSIS.md`(A-12.6), `A13_CONCURRENCY_VALIDATION.md`(A-12.7), `A13_IMPACT_MAPPING.md`(A-12.9)

## PART 1 — State Transition 전략 검토

### ① State 변경 위치: "Financial Workflow 이후 상태 변경"(현행) vs "상태 변경 이후 Financial Workflow"(가상)

| 기준 | 현행(Workflow → 상태 변경) | 가상(상태 변경 → Workflow) |
|---|---|---|
| 데이터 일관성 | "`open`인데 일부 금전 데이터가 이미 있는" 상태가 가능(A-12.6에서 확인된 실제 위험) | "`closed`인데 금전 데이터가 아직 없는" 상태가 가능 — **반대 방향의 불일치** |
| 실패 복구 | 재시도 가능(상태가 `open`으로 남음), 단 중복 위험(A-12.6) | 재시도하려면 "이미 `closed`인 세션의 Ledger를 다시 만든다"는 절차가 필요 — **ADR-006("Ledger는 한 번 쓰고 끝, 재계산 없음")과 충돌할 잠재 위험** |
| 동시 실행 | 확인(시작)과 변경(끝) 사이 간극이 가장 넓어 TOCTOU 윗도우가 큼(A-12.7) | 상태 변경이 먼저 일어나면 그 자체를 원자적으로 만들 경우 윗도우가 작아질 수 있음(②항목과 결합) |
| TOCTOU 위험 | 높음(A-12.7에서 확정) | 상태 변경 메커니즘에 따라 다름(①만으로는 답이 안 나옴 — ②와 결합해야 평가 가능) |
| Transaction 적용 용이성 | 가능(A-12.9 PART4에서 기술적으로 확인됨) | 가능하나 "상태 변경 실패 시 이후 단계를 진행하지 않는다"는 분기가 추가로 필요 |
| 현재 코드와의 적합성 | 100%(현재 그 자체) | 낮음 — 순서 자체를 뒤집는 구조 변경 |

**결정 가능한 것**: "상태 변경을 먼저 하는 방향은 ADR-006과 충돌할 잠재 위험이 있다"는 사실은 기존 ADR을 근거로 지금 판단할 수 있다. **결정 보류**: 어느 위치를 최종 채택할지는 새로운 순서 재설계에 해당하므로 본 문서에서 확정하지 않는다 — A-13 이후 검토 대상.

### ② State 변경 방식: 현재 방식 vs 조건부 UPDATE(`WHERE status='open'`) vs 기타(트랜잭션 내 재확인)

| 기준 | 현재 방식(확인 → 별도 쓰기) | 조건부 UPDATE(단일 SQL 문) | 트랜잭션 내 재확인(다중 문, 트랜잭션 콜백 안에서 재조회) |
|---|---|---|---|
| 데이터 일관성 | 낮음(TOCTOU 있음, A-12.7) | 높음(단일 문이라 원자적) | 트랜잭션 격리수준에 의존 — **미검증** |
| 실패 복구 | 가능하나 중복 위험 | 영향받은 행 수=0으로 즉시 판단 가능(단순) | 위와 동일하게 격리수준에 의존 |
| 동시 실행 | 취약(A-12.7에서 확정) | **SQLite 단일 쓰기자 모델 자체의 속성상, 단일 `UPDATE` 문은 그 자체로 동시 실행에 안전하다는 사실은 지금 확정 가능**(Drift의 트랜잭션 래핑 여부와 무관하게 SQLite가 보장) | Drift/SQLite의 기본 트랜잭션 모드(`BEGIN DEFERRED` vs `IMMEDIATE` 등)를 확인하지 않고는 안전성을 단정할 수 없음 |
| TOCTOU 위험 | 높음(확정) | 거의 없음(확인+쓰기가 한 문장) | 위와 동일하게 미검증 |
| Transaction 적용 용이성 | 가능(이미 확인) | 용이(단일 문이라 오히려 트랜잭션 없이도 일정 부분 보호됨) | 가능하나 안전성 검증이 먼저 필요 |
| 현재 코드와의 적합성 | 100% | 중간(`_requireSession()`의 역할 일부 대체, 순서 재배치 필요) | 낮음(검증되지 않은 동작에 의존) |

**결정 가능한 것**: ① "현재 방식은 TOCTOU에 노출된다"(A-12.7 재확인), ② "조건부 UPDATE 단일 문 자체는 SQLite 속성상 동시 실행에 안전하다"는 사실 — 둘 다 지금 확정할 수 있다. **결정 보류**: "트랜잭션 내 재확인" 방식의 실제 안전성(Drift/SQLite 격리수준 미검증 — 검증 자체가 코드를 실행해 확인하는 작업이라 본 문서(분석만, 코드 실행 없음) 범위 밖), 그리고 "조건부 UPDATE를 실제로 어떻게 코드에 채택할지"(순서 재배치 포함)는 A-13 이후 검토.

## PART 2 — Financial Workflow 경계 검토

| 단계 | 분류 | 근거 |
|---|---|---|
| **Settlement**(`payment_method_breakdowns` insert) | **① 반드시 Transaction 안** | `A13_FINANCIAL_WORKFLOW_STATE_ANALYSIS.md` PART5에서 이미 "4~8번(결제수단 insert~상태변경)은 반드시 같은 Transaction"으로 결론남 — 본 PART는 그 결론을 재확인할 뿐, 새로 판단하지 않음. |
| **StaffEarningLedger Snapshot** | **① 반드시 Transaction 안** | 위와 동일한 근거(같은 위험 구간, A-12.6 PART5). |
| **Receipt** | **② Transaction 밖에서도 가능** | A-12.5 PART1/PART3, A-12.6 PART5에서 일관되게 "DB 트랜잭션 밖의 외부 연동, 비동기/재시도 가능"으로 분류됨 — 미구현 상태이지만 그 성격(외부 부수효과) 자체는 바뀌지 않음. |
| **Sync** | **② Transaction 밖에서도 가능** | 위와 동일한 근거, 추가로 "현재 앱에 Sync 대상 자체가 없음"이 A-12.5/A-12.6/A-12.7에서 반복 확인됨. |

**판단 보류로 분류된 항목은 없다** — 4개 단계 모두 선행 문서에서 이미 일관되게 분류돼 있어, 본 PART에서 새로 판단을 미룰 이유가 없었다. (단, "Receipt/Sync를 정확히 언제·어떻게 트리거할지"는 별개의 구현 디테일이며, 그 디테일의 미결정 여부는 이 "경계 분류" 자체와는 무관하다.)

## PART 3 — Transaction Boundary 후보 정리(현재 코드 구조 기준)

| 항목 | 후보 |
|---|---|
| **Transaction 시작 시점** | Settlement 검증(`paidTotal == finalAmount`) 통과 직후 — 즉 현재 코드의 결제수단 insert(`_db.batch()`) 직전 지점. |
| **Transaction 종료 시점** | Session 상태 변경(`status='closed'` 갱신) 완료 직후. |
| **Transaction 내부 작업** | 결제수단 insert, 품목 조회(`StaffEarningEngine` 입력 준비), `StaffEarningEngine.calcEarnings()` 호출(순수 계산이라 트랜잭션 여부와 무관하지만 같은 스코프 안에 있음), Ledger insert(조건부), 상태 변경. |
| **Transaction 외부 작업** | 입력 검증(`paymentMethods` 형식), 세션 조회+가드(상태 확인 자체는 PART1 ①/②의 미결 사항과 연결되므로 경계가 완전히 확정된 것은 아님 — 다만 "현재 코드 그대로 둔다면" 트랜잭션 시작 이전에 위치), 마지막 재조회(이미 커밋된 데이터를 다시 읍는 것뿐이라 트랜잭션 안/밖 어디든 무방, 현재는 밖). |

**본 PART는 A-12.6 PART1(9단계 분해)과 A-12.9 PART4(같은 DB 인스턴스, 선례 확인)에서 이미 식별된 경계를 그대로 정리한 것이다 — 새로운 구현 방법을 제안하지 않는다.**

## PART 4 — TOCTOU 대응 방향 검토

> 4가지를 배타적 선택지가 아니라 독립적으로 분류한다(지시대로).

| 옵션 | 분류 | 근거 |
|---|---|---|
| **현재 구조 유지**(추가 조치 없음) | **A-13 MVP에서는 일시적으로 허용 가능 — 단, UI 연결 전 재검토 필수** | **결정적 근거**: `A13_IMPACT_MAPPING.md` PART1에서 확인된 사실 — `closeSession()`을 호출하는 production 코드(화면)가 **현재 0건**이다. 동시 실행(A-12.7의 핵심 위험)은 "겹치는 두 호출"이 전제인데, 호출자 자체가 없으므로 **이 위험은 지금 당장 노출되지 않는다.** 따라서 "당장 막아야 한다"는 긴급성은 낮다. 단, 이건 "막을 필요가 없다"는 뜻이 아니라 "지금은 노출이 없을 뿐"이라는 뜻이므로, UI가 이 메서드를 호출하게 되는 시점 이전에 재검토해야 한다. |
| **조건부 UPDATE** | **후속 단계에서 추가 가능** | PART1 ②에서 "단일 문 자체는 동시 실행에 안전하다"는 사실은 확정했으나, 실제 채택은 순서 재배치(PART1 ①)를 동반하는 구조 변경(`A13_IMPACT_MAPPING.md` PART6 "보통 작업")이라 MVP 필수로 보지 않는다 — 위 "현재 구조 유지" 항목의 근거(호출자 없음)와 같은 이유로 긴급성이 낮다. |
| **DB Constraint** | **판단 보류** | `A13_IMPACT_MAPPING.md` PART6에서 이미 "어려운 작업"으로 분류됐고, 특히 Settlement(`payment_method_breakdowns`)는 분할결제로 한 세션에 여러 정상 행이 존재할 수 있어 단순 UNIQUE 설계가 부적합할 가능성이 A-12.6에서 식별된 채 미해결 — 키 설계 자체가 아직 없어 "판단 보류"가 정확하다. |
| **Idempotency Key** | **판단 보류(단, 이 Workflow에서는 독립적인 4번째 옵션이 아님)** | `A13_CONCURRENCY_VALIDATION.md` PART2에서 이미 "`closeSession()`의 맥락에서 `sessionId`를 키로 쓰는 순간 DB Constraint 계층으로 귀결된다"고 분석됨 — 즉 본 Workflow에서는 DB Constraint와 같은 판단 보류 상태를 공유한다. (Receipt/Sync처럼 향후 별도 외부 연동이 생기면 그때는 독립적으로 유효한 개념이 될 수 있음 — 그 경우는 본 PART의 범위 밖.) |

## PART 5 — Failure Scenario 우선순위(기존 분석 재정리, 신규 시나리오 없음)

| 시나리오(출처) | 우선순위 | 구분 |
|---|---|---|
| 부분 실패로 인한 Settlement/Ledger 중복(순차 재시도 — 앱 종료/전원 종료/단계 실패 후 재호출, A-12.6 PART2·PART3) | **Critical** | **A-13에서 반드시 해결** — 단일 사용자·단일 호출만으로도(동시성 불필요) 발생 가능한 가장 기본적인 실패 모드이며, UI가 연결되는 즉시(멀티 호출 없이도) 실현 가능성이 높다. |
| 동시 실행으로 인한 중복(Thread A/B, UI 중복 클릭, A-12.7 PART1) | **High** | **후속 단계에서 해결 가능** — PART4에서 확정한 근거(현재 호출자 0건) 그대로, 지금 당장의 노출은 없으나 UI 연결 시 즉시 재평가 필요. |
| DB Write 실패(디스크 풀, 권한 오류 등, A-12.6 PART2) | **Medium** | **후속 단계에서 해결 가능** — SQLite 자체의 ACID 보장(단일 트랜잭션 단위)으로 이미 상당 부분 처리되는 영역이라(A-12.6에서 확인), 추가 조치의 우선순위가 낮음. |
| Receipt 중복 발급(미구현, A-12.5 PART5·A-12.6 PART4) | **Low** | **후속 단계에서 해결 가능** — 기능 자체가 아직 없고, 데이터 손상이 아니라 외부 부수효과(출력)의 문제. |
| Sync 중복 수행/네트워크 재전송(미구현, A-12.5 PART5·A-12.7 PART1) | **Low** | **후속 단계에서 해결 가능** — 기능 자체가 아직 없음(여러 문서에서 반복 확인). |

## PART 6 — A-13 MVP 범위 확정

| 구분 | 항목 |
|---|---|
| **A-13에서 반드시 구현** | `closeSession()`의 Settlement insert~Session 상태 변경 구간(PART3에서 정리한 경계)을 단일 Transaction으로 묶는다 — PART5의 Critical 시나리오(부분 실패로 인한 중복)를 해소하기 위한 최소 조치. |
| **A-13에서 구현하지 않음** | 조건부 UPDATE로의 전환(순서 재배치 포함), DB Constraint(UNIQUE 등) 추가, Idempotency Key 인프라, State Machine 변경(`CLOSING` 상태 추가 등), Workflow 계층(별도 클래스) 분리, Receipt/Sync 실제 구현. |
| **A-14 이후 검토** | 위 "구현하지 않음" 항목들 — 특히 `closeSession()`을 실제로 호출하는 UI 화면이 추가되는 시점에는 PART4/PART5에서 "후속 단계"로 분류한 동시성 대응(조건부 UPDATE, DB Constraint)을 **반드시 재평가**해야 한다(현재의 낮은 우선순위는 "호출자 없음"이라는 조건에 의존하므로, 그 조건이 사라지면 우선순위도 같이 바뀐다). |

## ADR 작성 여부

**작성한다.** PART6에서 "Transaction으로 묶는다"는 것이 A-12.5/A-12.6/A-12.9에서 반복적으로 "권장(미확정)"으로만 머물던 사안을 **A-13 MVP의 확정된 범위**로 전환시켰고, 이는 새로운 패턴을 발명한 것은 아니지만 "지금 무엇을 하고, 무엇을 미룰지"에 대한 실제 결정이다 — `docs/adr/ADR-007-a13-mvp-transaction-scope.md`로 별도 작성한다(본 지시문은 "ADR 수정 금지"이지 "신규 ADR 작성 금지"가 아니며, 결과물 절에서 "실제로 새로운 설계 결정이 확정된 경우에만 작성 여부를 판단"하라고 명시했으므로 이 결정은 그 조건을 충족한다).
