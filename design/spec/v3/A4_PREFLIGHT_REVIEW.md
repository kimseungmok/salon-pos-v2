# A-4 구현 전 분석 — 직원 재직/퇴사 상태 (컬럼 추가 없이)

> **기준 문서**: `MVP_IMPACT_MAP.md` §A-4, `DESIGN_REVIEW.md` §5, `PRIORITIZATION.md` A-4
> **전제 변경**: 기존 문서들은 "상태 필드를 추가"한다고 전제했으나, 본 분석은 **컬럼 추가가 금지된 조건**에서 무엇이 가능한지를 다시 검토한다. 즉 `MVP_IMPACT_MAP.md`가 "테이블 변경이 필요한 사안"으로 분류했던 것을, **기존 컬럼 재활용으로 해결 가능한지**를 먼저 확인하는 것이 본 문서의 핵심 질문이다.
> **범위 제한**: 설계 문서만 작성. 코드/SQL 생성, 테이블/컬럼 추가 없음.
> 작성일: 2026-06-23

## 핵심 발견 (먼저 결론)

`Staff.accountStatus`는 **이미 nullable TEXT 컬럼**이며, 현재 `null`(미초대)/`'待機中'`(초대중)/`'連結済み'`(연결완료) 3가지 값만 쓰고 있다. **새 컬럼을 추가하지 않고, 이 필드에 4번째 값(예: `'退職済み'`)을 추가하는 것만으로 "재직/퇴사 상태"를 표현할 수 있다.** 이는 `groupOf()`가 하드코딩된 임계값을 쓰는 패턴, `PrepaidPassMenu.status(active/disabled)`가 이미 같은 컬럼 하나로 상태를 표현하는 패턴과 동일한 방식이라 기존 코드 철학과도 일치한다. 아래 분석은 이 접근(기존 컬럼 재활용)을 전제로 한다.

## 4. 현재 delete 기반 구조가 status 기반으로 변경될 경우 영향 범위

| 항목 | delete 기반(현재) | status 기반(변경 후) |
|---|---|---|
| `Shift` cascade | 퇴사 시 해당 직원의 모든 시프트 기록이 **함께 영구 삭제됨** | 더 이상 발동하지 않음 — 과거 시프트 기록이 보존됨(부수적 개선) |
| 다른 모듈의 `staffId` 참조 | FK 제약이 없어 삭제돼도 에러는 안 나지만, 참조값이 가리키는 행이 영구히 사라져 **이름을 표시할 수 없게 됨**(고아 참조) | `Staff` 행 자체가 남아있으므로 이름/연락처 등 과거 데이터를 그대로 조회 가능 — `DESIGN_REVIEW.md` §5가 지적한 문제가 정확히 해소됨 |
| `watchStaff()` 목록 조회 | 삭제된 행은 자동으로 목록에서 빠짐(쿼리 단순) | **퇴사자가 계속 행으로 남아있어, 화면에서 걸러내는 책임이 쿼리 또는 화면으로 옮겨감** — 지금 `watchStaff()`는 `accountStatus`로 필터링하지 않으므로, 그대로 두면 퇴사자가 직원목록에 계속 보이게 됨(§6 리스크) |
| 전화번호 재사용 검증(`inviteStaff()`의 중복체크) | 삭제된 사람의 번호는 즉시 재사용 가능 | 퇴사자의 번호가 `Staff` 행에 남아있으므로, **퇴사자와 같은 번호로 신규 직원을 등록하려 하면 현재 중복체크 로직이 막아버림**(새로운 부작용 — §6에서 다룸) |

## 5. A-1(recordVisit) 및 A-2(예약완료)에 미치는 영향

### A-1(`PaymentRepository.pay()` → `recordVisit()`)
- `_staffIdOf()`는 `OrderItem.staffId`를 그대로 읽어 `recordVisit()`에 전달할 뿐, `Staff` 테이블의 상태를 조회하지 않는다. **A-4가 구현돼도 A-1의 동작은 전혀 바뀌지 않는다** — 퇴사한 직원이 과거에 담당한 `OrderItem`이 있어도 그 `staffId` 문자열은 그대로 유효하게 전달됨(코드 변경 불필요).

### A-2(`BookingRepository.completeBooking()`)
- `completeBooking()`은 `Booking.status`만 갱신하고 `staffId`를 전혀 다루지 않는다 — **A-4와 직접적인 상호작용이 없다.**
- 다만 `createBooking()`(A-2가 아니라 기존 M4 기능)이 호출하는 `_assertStaffAvailable()` → `isOnShift()` 경로는 **퇴사한 직원에게도 여전히 새 예약을 받아줄 수 있다**(상태값을 검사하지 않으므로) — 이는 A-2 자체의 문제가 아니라 booking 모듈 전반의 문제이며, §6에서 별도 업무규칙으로 다룬다.

**결론**: A-1/A-2 코드는 **수정 없이 그대로 동작**한다. 영향은 "A-4 이후에 새로 결정해야 할 booking 쪽 정책"으로만 남는다.

## 7. 마이그레이션 필요 여부 (테이블 추가 금지 조건 유지)

**불필요.** `accountStatus`는 Drift에서 `text().nullable()()`로 선언된 **자유 텍스트 컬럼**이라, 허용 가능한 값의 집합은 스키마가 아니라 애플리케이션 코드(검증 로직)가 정한다. 새 값(`'退職済み'`)을 쓰는 것은 컬럼 정의를 바꾸는 것이 아니라 **그 컬럼에 들어가는 문자열 하나를 추가하는 것**이므로, Drift 스키마 버전(`schemaVersion`)을 올리거나 마이그레이션을 작성할 필요가 없다 — 기존 DB 파일을 가진 사용자도 앱 업데이트만으로 그대로 동작한다.

## 종합 — 출력 형식

### 충분한 부분
- `Staff.accountStatus`가 이미 nullable TEXT라 **상태값 추가에 스키마 변경이 전혀 필요 없다** — 마이그레이션 비용 0
- A-1/A-2 코드는 `Staff`의 상태를 조회하지 않는 구조라 **A-4 구현이 두 기능에 어떤 회귀도 일으키지 않는다**(§5 확인)
- `Shift`의 `KeyAction.cascade`가 하드삭제를 멈추는 순간 자동으로 무력화되어, **추가 코드 없이 시프트 이력 손실 문제가 같이 해결됨**(부수적 이득)
- 기존 `removeStaff()` 테스트가 "정상 삭제" 케이스를 검증하지 않고 있어, 동작을 바꿔도 **기존 테스트가 깨질 위험이 없음**

### 부족한 부분
- `watchStaff()`가 `accountStatus`로 필터링하지 않아, **퇴사자가 그대로 직원목록에 남아 화면에 계속 보임** — 화면(33번, 향후 11번)이 직접 걸러내야 하는데 그 책임 소재가 아직 없음
- `inviteStaff()`의 전화번호 중복체크가 "퇴사자가 쓰던 번호"까지 막아버리는 새로운 부작용이 생김 — 현재 중복체크는 상태를 구분하지 않고 전화번호만 봄
- `BookingRepository._assertStaffAvailable()`이 퇴사자에게도 새 예약을 계속 허용한다 — A-4만으로는 해결되지 않는 인접 영역(§6에서 별도 결정 필요로 남김)
- `accountStatus`라는 이름 자체가 "계정연결 상태"를 뜻하는데 "재직 상태"라는 다른 의미를 같은 필드에 같이 싣는 것이라, **필드명과 실제 의미가 어긋나는 설계 부채**가 생김(컬럼 추가 금지 조건의 트레이드오프)

### 리스크
- **의미 혼용 리스크**: `accountStatus`가 "초대 진행상태"와 "재직 상태"라는 서로 다른 두 축을 한 필드에 담게 되어, 향후 "초대는 됐는데 아직 재직 여부가 불분명한" 케이스(예: 재초대 대상)가 생기면 값 설계가 더 복잡해질 수 있음
- **화면 필터링 누락 리스크**: §"부족한 부분" 1번을 그대로 두고 화면만 구현하면, 퇴사자가 직원 선택 드롭다운(예약 화면의 담당자 선택 등)에 계속 나타나 현장에서 혼란을 줄 수 있음
- **전화번호 재사용 차단 리스크**: 실제로 퇴사자가 쓰던 번호로 새 직원이 들어오는 일이 드물지 않은데, 현재 로직대로면 등록이 막혀 직원이 임의로 다른 번호를 입력하는 우회가 생길 수 있음(고객 모듈에서 이미 한 차례 식별된 것과 같은 종류의 문제)

### 권장 구현 방식
1. `removeStaff()`의 내부 구현을 `_db.delete()` → `_db.update(... accountStatus: '退職済み')`로 교체(메서드 시그니처는 유지)
2. `watchStaff()`에 `'退職済み'`를 제외하는 필터를 추가하거나, "전체 보기" 옵션을 화면에 별도로 제공할지 결정 — 본 분석은 어느 쪽이 맞는지 결론내지 않고 **결정이 필요하다는 사실만 표시**(`BUSINESS_RULES_TO_DECIDE.md`에 신규 항목으로 추가할 가치가 있음)
3. `inviteStaff()`의 중복체크가 퇴사자 번호를 막는 문제는, "퇴사자와 같은 번호 등록 시도시 안내 메시지를 다르게 보여줄지"를 별도 업무규칙으로 분리해 결정
4. `BookingRepository._assertStaffAvailable()`의 퇴사자 예약 차단 여부는 A-4의 1차 범위에 포함하지 않고, **별도 후속 결정 사안으로 명시적으로 분리**(A-3 예약변경 작업과 함께 검토하는 것이 효율적 — 두 작업 모두 `_assertStaffAvailable()`을 만지게 됨)
5. 마이그레이션 작업 없음 — 구현 착수 시 스키마 버전업/migration 단계를 별도로 계획할 필요가 없다는 점을 구현 체크리스트에 명시
