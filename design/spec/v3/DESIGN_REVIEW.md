# 설계 리뷰 — 실제 살롱 운영 업무 기준

> **기준 문서**: `SCREENS_ERD_TABLES.md`(21개 테이블, 9개 모듈)
> **검토 방법**: 테이블 정의뿐 아니라 실제 레포지토리 로직(`lib/features/*/data/*_repository.dart`)까지 다시 읽어서, "컬럼이 있다"가 아니라 "그 컬럼을 실제로 채우는 코드 경로가 있는가"를 기준으로 판단했다.
> **목적**: 기존 설계를 바꾸는 게 아니라 **누락된 운영 프로세스와 확장 위험을 찾는 것**. 이 문서 자체는 테이블 추가/수정을 하지 않으며, 코드·SQL도 생성하지 않는다.
> 작성일: 2026-06-23

## 1. 예약(Reservations)

### 현재 설계로 충분한 항목
- **예약 생성**: `createBooking()` — 고객 필수, 메뉴 1개 이상 필수, 종료시각 검증, 예약금 ON 시 금액 검증까지 입력단계 검증 충분
- **직원 배정(생성 시점)**: `_assertStaffAvailable()`가 ①해당 시프트 휴무 여부 ②같은 담당자의 기존 예약과 시간 중복 여부를 사전에 차단 — 더블부킹 방지 로직은 견고함
- **노쇼 처리(예약금 관점)**: `cancelBooking(reason: 'customer_late_or_noshow')` → `status='noshow'` + `depositRefunded=false` 규칙이 F-BOOK-04 그대로 정확히 구현됨
- **상품/시술과의 연결(단순 케이스)**: `productIdsCsv`로 N개 메뉴 연결 가능, `booking_logic.dart`의 `computeEndAt()`이 메뉴별 `durationMin` 합산으로 종료시각 자동 산출

### 현재 설계에서 부족한 항목
| 항목 | 내용 |
|---|---|
| **예약 변경(수정) 프로세스 자체가 없음** | `BookingRepository`에 `createBooking()`/`cancelBooking()`만 있고 **`updateBooking()`이 존재하지 않는다.** 시간 변경, 담당자 재배정, 메뉴 추가/제거를 하려면 현재 구조상 "취소 후 재생성"밖에 방법이 없음 — 이 경우 예약금 상태(이미 받은 돈)가 새 예약으로 이어지지 않고 끊어짐 |
| **방문 완료 처리가 정의되어 있지 않음** | `Booking.status`에 `'completed'`값이 주석으로 예고되어 있지만(`confirmed/completed/noshow/cancelled`), **이 상태로 전환시키는 함수가 어디에도 없다.** `createBooking()`은 `'confirmed'`로 시작하고, `cancelBooking()`은 `cancelled`/`noshow`로만 보낸다 |
| **예약↔방문이력(VisitRecord) 연결이 끊어져 있음** | `customer/data_spec.md`가 전제하는 F-CUST-01 그룹 자동분류(常連/休眠ぎみ)는 `VisitRecord`를 근거로 계산되는데, `CustomerRepository.recordVisit()`을 **호출하는 코드가 코드베이스 전체에 단 한 곳도 없다**(grep 결과 정의부만 존재). 즉 예약이 "완료"되어도 자동으로 방문 1건이 쌓이지 않으므로, **그룹 자동분류 기능 자체가 현재 운영 데이터 흐름상 작동하지 않는다** |
| **예약↔결제(Payment/Order) 연결 고리가 없음** | `Booking` 테이블과 `Order`/`Payment` 테이블 사이에 **어떤 컬럼도 서로를 참조하지 않는다**(`Order.bookingId` 같은 필드 없음). 예약된 메뉴를 그대로 결제로 넘기는 자동 연계가 불가능 — 직원이 예약 내용을 보고 02번 화면에서 **수동으로 똑같은 메뉴를 다시 담아야** 함(입력 이중화, 실수 위험) |
| **직원 재배정(예약 생성 이후)** | `staffId`는 생성 시점에만 설정되고, 이후 변경하는 메서드가 없음(위 "예약 변경" 항목과 동일 원인) |
| **반복예약(`repeatRule`) 실제 처리 로직 없음** | 컬럼은 있으나(`none/weekly/...`), 실제로 N건을 일괄 생성하는 로직이 레포지토리에 없음 — 값을 저장은 하지만 동작은 안 함 |

### 향후 확장 시 위험 요소
- **"취소 후 재생성" 임시방편이 관례화될 위험**: 변경 기능이 없으면 현장 직원이 "시간 바꿔주세요" 요청을 처리할 방법이 없어, 결국 취소(노쇼 아닌 일반취소로 처리)→재생성을 반복하게 됨 → `cancelled` 레코드가 비정상적으로 누적되어 **취소율 통계(향후 売上レポート 보너스탭)가 왜곡됨**
- **예약-결제 이중입력은 매장이 바쁠 때 누락·오류로 직결**: 직원이 예약을 보고 결제창에서 다른 메뉴/가격을 입력해도 시스템이 막을 방법이 없음(검증 로직이 둘을 비교하지 않음)
- **고객 그룹 자동분류가 "보이지만 작동하지 않는" 상태로 운영될 위험**: 09번 화면은 이미 구현되어 그룹 탭이 화면에 나타나지만, 실제로는 `VisitRecord`가 쌓이지 않아 **모든 고객이 영원히 初回来店 상태로 보일 가능성이 높음** — 가장 시급한 위험

### 권장 개선안 (테이블 변경 없이 가능한 범위)
1. 방문 완료 처리 시점(체크아웃 완료, 또는 결제 완료 시점)에 **기존 `CustomerRepository.recordVisit()`을 호출하는 연결 코드**를 추가 — 새 테이블 없이 이미 있는 메서드를 "누가 호출하느냐"만 정하면 해결됨
2. `Booking` 변경은 신규 테이블 없이 **기존 `Bookings` 테이블에 대한 update 메서드(`updateBooking()`)를 레포지토리에 추가**하는 것만으로 처리 가능(테이블 구조 변경 없음, 로직만 추가) — 단, 이번 리뷰는 변경 금지 범위라 "필요하다"는 사실만 기록
3. 예약↔결제 연결은 테이블을 늘리지 않고도, **02번 화면에서 결제 시작 시 직전 예약의 메뉴 목록을 카트에 미리 채워주는 화면단 로직**으로 1차 완화 가능(데이터베이스 레벨 FK 없이도 화면 흐름으로 봉합)
4. 반복예약은 "값은 저장하되 동작 없음" 상태를 화면에 솔직하게 노출(예: "반복예약은 준비 중입니다" 안내)해 기능이 있는 것처럼 오인되는 걸 방지

## 2. 고객(Customers)

### 현재 설계로 충분한 항목
- **신규 고객 등록**: `createCustomer()` — 이름/전화번호 필수, 전화번호 중복 차단(기존 고객 이름까지 안내) 양호
- **재방문 고객 조회**: `search()` — 이름 부분일치/전화번호 부분일치 검색 가능
- **고객 메모**: `updateMemo()` — F-CUST-02 그대로, 오너 권한 체크는 화면단 책임으로 명확히 분리되어 있음(설계 의도가 문서화됨)
- **방문 이력 조회(데이터 구조 자체)**: `VisitRecord` 테이블 자체는 customerId/visitDate/staffId/amount/status를 다 갖추고 있어, **데이터가 쌓이기만 하면** 조회는 문제없음

### 현재 설계에서 부족한 항목
| 항목 | 내용 |
|---|---|
| **방문 이력이 실제로 쌓이지 않음(§1과 동일 근본원인)** | "재방문 고객 조회"는 가능하지만, 그 판단 근거인 `VisitRecord`가 자동 적재되지 않으므로 **실질적으로는 빈 이력만 조회하게 됨** |
| **고객 정보 수정(이름/전화번호) 기능이 없음** | `createCustomer()`만 있고 `updateCustomer()`가 없음 — 등록 시 오타가 나면 고칠 방법이 없음(메모만 수정 가능) |
| **고객 삭제/비활성화 기능이 없음** | 탈퇴/정보삭제 요청에 대응하는 메서드가 없음(개인정보 보호 관점에서 향후 요구될 가능성) |

### 향후 확장 시 위험 요소
- **그룹 자동분류(常連/休眠ぎみ) 기능이 마케팅(쿠폰/캠페인) 타겟팅과 결합될 미래 계획과 충돌**: `VisitRecord`가 안 쌓이면 "휴면고객 대상 쿠폰" 같은 향후 확장 자체가 토대부터 흔들림
- **개인정보 삭제 요청 대응 불가**: 일본 개인정보보호법 대응 시 "삭제 권리" 요청이 오면 현재 구조로는 수동 DB 조작 외에 대응수단이 없음

### 권장 개선안
1. §1의 권장사항 1번(방문완료 시 `recordVisit()` 연결)이 그대로 본 항목의 핵심 해결책 — 동일 이슈이므로 중복 작업 불필요
2. `updateCustomer()`는 기존 `Customers` 테이블 구조를 전혀 바꾸지 않고 **레포지토리 메서드만 추가**하면 되는 사안 — 본 리뷰 범위(변경 금지) 밖이라 필요성만 기록

## 3. 재고(Products / Inventory)

### 현재 설계로 충분한 항목
- **입고(stock_in)/사용(use)/폐기(disposal)/조정(adjustment) 4종 사유**가 `adjustQuantity()`에 명확히 검증되고, 모든 변경이 `InventoryLog`에 **자동으로** 기록됨(이 부분은 booking/customer와 달리 "기록 함수가 실제로 호출되는 경로"가 코드 안에 존재 — 호출자가 `adjustQuantity()` 자기 자신이라 누락 위험 없음)
- **상품(Product)과 재고(Inventory)를 의도적으로 분리**한 설계(F-INV-00)는 "판매상품 vs 소모자재"가 다른 살롱 운영 현실에 맞고, 잘못 합쳤다가 풀어야 하는 향후 리스크를 미리 차단한 합리적 결정으로 보임

### 현재 설계에서 부족한 항목
| 항목 | 내용 |
|---|---|
| **재고조사(棚卸)를 위한 전용 흐름이 없음** | "조사일 시스템수량 vs 실사수량 차이"를 한 이벤트로 기록하는 구조가 없다. `adjustment` 사유로 `delta`만 남기면, **조사 당시의 "이전 수량/실사 수량/담당자/조사일시"가 한 묶음으로 보존되지 않고** 단순 증감값만 남음 — `CashCount`(개점/폐점)는 `totalAmount`/`expectedAmount`/`diffAmount`를 한 레코드에 같이 저장하는 것과 비교하면 **재고는 그 수준의 구조화가 안 되어 있음** |
| **품목 삭제 시 변동 이력이 함께 사라짐** | `InventoryLog.itemId`가 `KeyAction.cascade`로 `InventoryItem`을 참조 — **품목을 삭제(`deleteItem()`)하면 그 품목의 모든 입출고 이력이 통째로 사라진다.** 회계/세무 관점에서 소모품이라도 사용 이력이 영구 삭제되는 건 감사 추적성(audit trail) 측면에서 위험 |
| **품목 비활성화(판매중단/폐기예정) 개념이 없음** | `PrepaidPassMenu`에는 `status(active/disabled)`가 있는데 `InventoryItem`에는 동급 필드가 없음 — 더 이상 안 쓰는 자재를 "삭제"(이력 소실)로만 처리할 수 있음 |

### 향후 확장 시 위험 요소
- **세무조사/내부감사 시 "재고가 왜 줄었는지" 역추적이 불가능해질 위험**: 품목을 삭제한 적이 있으면 그 품목의 입출고 이력 전체가 사라진 상태이므로 설명할 근거 자체가 없음
- **재고조사 결과를 정기적으로 남기는 운영 관행이 자리잡기 어려움**: 구조적으로 "조사" 이벤트가 "조정" 이벤트와 구분되지 않아, 나중에 "지난달 재고조사 결과가 뭐였지?"를 조회할 방법이 없음

### 권장 개선안
1. 품목 삭제는 하드 삭제(`deleteItem`) 대신 **비활성화 플래그를 쓰는 운영 정책으로 전환**(테이블 변경 없이도, 기존 `category` 필드에 "(廃止)" 접두어를 붙이는 임시 운영 규칙 정도는 가능하나 임시방편일 뿐 — 본 리뷰는 변경을 권고하지 않고 위험만 기록)
2. 재고조사는 현재도 `reason='adjustment'`로 기록 자체는 가능하므로, **운영 매뉴얼 차원에서 "조사 시 메모에 조사일자를 남긴다" 같은 절차적 보완**으로 1차 완화 가능(테이블 변경 없는 임시 대응)

## 4. 결제(Payments)

### 현재 설계로 충분한 항목
- **현금**: `cashReceived`/`cashChange` 필드와 `computeChange()` 로직으로 거스름돈까지 정확히 계산·저장됨, 받은금액 < 결제금액 시 검증으로 차단
- **카드**: `method='card'`로 기록 가능(단말기 연동은 범위 외로 명시되어 있음 — 모킹 처리가 설계 의도와 일치)
- **QR결제(PayPay/LINE Pay)**: `_validMethods`에 `paypay`/`linepay`가 이미 포함되어 결제 기록 자체는 가능
- **분할결제/취소**: 잔여금액 계산(`remainingAmount`), 초과결제 차단, 취소 시 트랜잭션 기반 원자적 처리(포인트 환원+선불권 환원+상태변경이 한 트랜잭션) — 이 부분은 설계 수준이 높음

### 현재 설계에서 부족한 항목
| 항목 | 내용 |
|---|---|
| **QR결제의 외부 결제망 대조용 식별자가 없음** | `Payment` 테이블에 `method`/`amount`만 있고, PayPay·LINE Pay 같은 **외부 결제사의 거래번호(승인번호/거래ID)를 저장할 컬럼이 없다.** 모킹 단계에서는 문제없지만, 실제 QR결제 연동 시 "이 결제가 실제로 PayPay 쪽에서도 승인됐는지" 대조(정산 reconciliation)가 불가능 |
| **결제수단별 부분환불 단위가 "주문 전체"뿐** | `cancelOrder()`는 주문에 속한 **모든 Payment를 한꺼번에 refunded 처리**한다. 분할결제로 현금+카드를 나눠 받은 주문에서 "카드 결제분만 취소"하는 부분 취소는 현재 불가능 |

### 향후 확장 시 위험 요소
- **실제 QR결제 PG 연동 시점에 식별자 컬럼 부재로 정산 불일치 발생 가능**: 지금은 자체 기록만 믿고 있어, 결제사 쪽 매출과 앱 쪽 매출이 어긋나도 추적할 단서가 없음
- **분할결제 + 부분환불 조합 케이스가 늘어날 살롱 특성상(예약금 일부만 환불 등) 향후 요구가 들어올 가능성 높음** — 지금 구조로는 "전체 취소"만 가능해 운영상 불편이 누적될 수 있음

### 권장 개선안
1. 외부 결제식별자는 현재 `Payment` 테이블 컬럼을 늘리지 않고는 저장할 곳이 없음 — **실제 PG 연동 착수 전에 반드시 재검토가 필요하다는 점만 기록**(이번 리뷰에서는 추가하지 않음)
2. 부분환불은 당장 급하지 않다면 "주문을 분할결제 단위로 더 잘게 쪼개서 별도 Order로 만드는" 운영 관행으로 우회 가능(테이블 변경 없는 임시 대응, 다만 메뉴별결제 UI가 아직 없어 현실적으로는 차후 과제)

## 5. 직원(Staffs)

### 현재 설계로 충분한 항목
- **담당자 지정(예약 생성 시점)**: §1에서 다룬 대로 가용여부 검증까지 포함해 견고함
- **초대 플로우**: 待機中→連結済み 상태 전환, 재전송/삭제 모두 구현되어 F-STAFF-01 요건 충족

### 현재 설계에서 부족한 항목
| 항목 | 내용 |
|---|---|
| **로그인(PIN) 처리 주체가 전혀 정의되어 있지 않음** | `Staff` 테이블에 PIN/인증 관련 컬럼이 의도적으로 없는 것은 F-STAFF-00 설계 결정(외부 통근관리시스템 위임)이지만, **그 외부 시스템과 `Staff.id`를 어떻게 매칭할지에 대한 연동 키조차 없다.** `01_login_main` 화면(PIN 6자리)이 목업으로 존재하는데, 로그인 성공 시 "지금 로그인한 사람이 누구인지" 앱이 알아낼 방법이 현재 설계엔 전혀 없음 |
| **재직/퇴사 상태 필드가 없음 — `removeStaff()`가 하드 삭제** | `Staff` 테이블에 `status(active/resigned)` 같은 필드가 없고, 퇴사 처리는 `removeStaff()`로 **행 자체를 영구 삭제**한다. `VisitRecord.staffId`/`OrderItem.staffId`/`Booking.staffId`/`InventoryLog.staffId`/`CashCount.confirmedBy`가 전부 FK 제약 없는 텍스트 참조이므로, **퇴사 직원을 삭제하는 순간 그 직원이 담당했던 모든 과거 매출·예약·재고기록에서 이름을 더 이상 표시할 수 없게 됨**(고아 참조) |

### 향후 확장 시 위험 요소
- **로그인 기능 구현 착수 시 설계 공백이 가장 먼저 드러날 영역**: 01번 화면을 실제로 만드는 순간 "PIN을 어디서 검증하고 누구로 식별하는가"를 처음부터 다시 설계해야 함 — 지금 미리 인지해두지 않으면 다른 모듈(특히 결제승인자, 시재확정자)과 한꺼번에 얽혀 더 큰 재작업이 될 위험
- **퇴사자가 1명만 생겨도 売上レポート의 "スタッフ実績"(보너스 영역, F-SALES-02) 집계가 망가질 위험**: 퇴사 직원의 이름이 사라지면 과거 매출 집계에서 "이름 없음" 또는 빈 참조로 나타나, 매장 운영진이 과거 실적을 신뢰하기 어려워짐
- **현재 ja_toss100 목업(11/12/13)에는 재직상태 개념이 화면상으로도 없어, 화면-데이터 양쪽에서 동시에 빠진 상태** — 한쪽만 고치면 불일치가 더 커짐

### 권장 개선안
1. 로그인 연동 방식(① 외부 시스템 API로 PIN검증 후 결과만 받는 방식 vs ② 이 앱 내부에 PIN을 별도로 저장·검증하는 방식)에 대한 **의사결정이 구현 착수 전 반드시 필요** — 결정에 따라 필요한 구조가 완전히 달라지므로 지금 임의로 정하지 않는 것이 맞음(이번 리뷰는 결정하지 않고 "결정이 필요하다"는 사실만 기록)
2. 퇴사 처리는 `removeStaff()`(하드 삭제) 대신 **상태 전환 방식으로 바꿀 필요성**이 있다는 점만 기록 — 테이블 구조 변경(필드 추가)이 필요한 사안이라 이번 리뷰에서는 실행하지 않음

## 종합 — 영역을 가로지르는 단일 근본원인

§1과 §2에서 각각 지적한 "그룹 자동분류가 작동하지 않는다"는 사실 하나가 두 영역에 동시에 영향을 준다. **"방문 완료" 이벤트가 예약/결제 어느 쪽에서도 `VisitRecord`로 이어지지 않는다는 점이 이번 리뷰에서 발견한 가장 우선순위 높은 단일 이슈**다 — 새 테이블 없이 기존 `recordVisit()` 호출 위치만 정하면 해소되는 사안이라, 구조 변경 없이도 가장 먼저 다룰 가치가 있다는 점을 강조해둔다.
