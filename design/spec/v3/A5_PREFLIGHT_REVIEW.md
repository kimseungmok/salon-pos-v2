# A-5 구현 전 프리플라이트 검토 — 재고 이력 보존 (컬럼 추가 없이)

> **기준 문서**: `MVP_IMPACT_MAP.md` §A-5, `DESIGN_REVIEW.md` §3, `A4_PREFLIGHT_REVIEW.md`(동일 패턴 — 컬럼 추가 없이 delete→상태기반 전환 가능 여부를 먼저 확인하는 접근)
> **용어 정정**: 질문에 쓰인 "Products / InventoryTransactions"는 실제 코드의 `InventoryItems`/`InventoryLogs`에 대응한다. `Products`(상품)와 `InventoryItems`(소모성 자재)는 **F-INV-00 원칙에 따라 의도적으로 완전히 분리**되어 있으며, 어떤 코드 레벨 참조도 없다(`grep` 결과 0건, 본 점검에서 재확인) — 이 분리는 버그가 아니라 명시적 설계 결정이므로, 아래 분석에서 "POS 결제 흐름과의 연결"은 **현재 존재하지 않는 연결을 가정 분석**하는 것임을 먼저 밝힌다.
> **범위 제한**: 설계 문서만 작성. 코드/SQL 생성·수정, 테이블/컬럼 추가 없음.
> 작성일: 2026-06-23

## 1. 현재 구조에서 입고/사용/폐기/재고조사(棚卸) 흐름이 가능한지

| 흐름 | 현재 가능 여부 | 근거 |
|---|---|---|
| 입고(stock_in) | ✅ 가능 | `adjustQuantity(reason: 'stock_in', delta: 양수)` — `InventoryLog`에 자동 기록 |
| 사용(use) | ✅ 가능 | `adjustQuantity(reason: 'use', delta: 음수)` — 동일 |
| 폐기(disposal) | ✅ 가능 | `adjustQuantity(reason: 'disposal', delta: 음수)` — 동일 |
| **재고조사(棚卸)** | **⚠️ 형식적으로만 가능, 구조적으로는 불완전** | `reason: 'adjustment'`로 임의의 `delta`(양/음 모두)를 기록할 수는 있으나, **"조사 시점의 시스템수량/실사수량/차이"를 한 묶음으로 남기는 구조가 없다.** `CashCount`가 `totalAmount`/`expectedAmount`/`diffAmount`를 한 레코드에 같이 저장하는 것과 달리, `InventoryLog`는 `delta`(차이값) 하나만 남고 "조사 당시 실제로 몇 개였는지"는 별도로 계산해서 메모해두지 않으면 사라짐 — `DESIGN_REVIEW.md` §3에서 이미 식별된 격차가 그대로 유지됨 |

**A-5에 대한 함의**: A-5("이력 보존")는 §1의 4개 흐름 중 **입고/사용/폐기는 이미 이력이 남고 있어 "보존"이 문제없이 되는 영역**이고, 문제는 ① 품목 삭제 시 그 이력이 cascade로 함께 사라지는 것(§4), ② 재고조사 자체가 구조적으로 약한 것(별도 사안, 본 문서 §7에서 범위 밖으로 분리)이다.

## 2. "현재 수량"과 "이력 기반 수량"이 어떻게 계산되는가

- **현재 수량**: `InventoryItem.quantity`라는 **저장된 단일 값**이다. `adjustQuantity()`가 호출될 때마다 `item.quantity + delta`를 계산해 그 결과를 덮어쓴다 — 즉 "현재 수량"은 매번 새로 계산되는 것이 아니라 **누적 갱신되는 캐시값**이다.
- **이력 기반 수량**: 코드 어디에도 `InventoryLog.delta`를 모두 합산해서 현재 수량을 "재계산"하는 함수가 없다. `InventoryLog`는 순수하게 **감사 추적용 기록**일 뿐, 현재 수량의 근거(source of truth)로 쓰이지 않는다.
- **두 값이 서로 다른 소스**라는 사실 자체가 구조적 위험이다: `InventoryItem.quantity`(저장값)와 `SUM(InventoryLog.delta)`(이력 합산값)가 항상 같아야 한다는 보장이 코드 레벨에 전혀 없다 — 한쪽만 갱신되고 다른 쪽이 누락되는 경로가 생기면(§5의 동시성 문제 등) **두 값이 조용히 어긋나기 시작하고, 이를 감지할 검증 로직이 없다.**

## 3. POS 결제 흐름(pay → recordVisit 이후)과 재고 차감 연결 방식

**현재 연결이 전혀 없다.** `PaymentRepository.pay()`/`createOrder()`는 `InventoryRepository`를 import하지도, 참조하지도 않는다(F-INV-00 — 의도적 미연동, 본 점검에서 재확인). 따라서:

- 결제가 완료되어 `recordVisit()`이 호출돼도(A-1), 그 주문에 포함된 상품이 소모한 자재 재고는 **자동으로 차감되지 않는다.**
- 재고 차감은 오직 직원이 14번 화면에서 **수동으로** `adjustQuantity(reason: 'use')`를 호출해야만 일어난다.

**이것이 A-5의 범위인가?** 아니다 — A-5는 "이미 기록되는 이력을 어떻게 보존할지"를 다루는 항목이고, "결제와 재고를 연동할지"는 F-INV-00이라는 **더 상위의, 이미 결정된 설계 원칙**을 뒤집는 별개의 의사결정이다. 본 점검은 이 연결의 부재를 "버그"가 아니라 **현재 전제**로 취급하고, A-5 분석은 이 전제 위에서 진행한다. (연동 자체를 재검토하려면 `BUSINESS_RULES_TO_DECIDE.md`에 새 항목으로 등재하는 것이 적절하며, 본 문서의 범위를 넘는다.)

## 4. 재고 음수 발생 가능성 및 방지 방식

- `adjustQuantity()`는 `newQuantity < 0`이면 `BusinessRuleException`으로 차단한다 — **애플리케이션 레벨 검증**이며, DB 제약(`CHECK` 등)으로 강제되는 것은 아니다.
- 이 검증은 **읽기(현재 `quantity` 조회)와 쓰기(갱신) 사이에 시간차가 있는 비원자적 검증**이다 — §5에서 다루는 동시성 문제와 직결된다. 두 요청이 동시에 들어오면 **둘 다 "차감 후에도 0 이상"이라고 각자 판단**해 통과시킬 수 있고, 그 결과 실제로는 음수가 될 수 있다(개별 검증은 정상 동작했지만 결합 결과가 잘못되는 전형적인 TOCTOU 패턴).
- 즉 "음수 방지 로직이 있다"와 "음수가 절대 발생하지 않는다"는 동일하지 않다 — **현재 구조에서는 동시 요청 상황에서 음수가 실제로 발생할 수 있다.**

## 5. 동일 상품 동시 결제 시 race condition 가능성

`adjustQuantity()`의 구현을 그대로 보면:

1. `item`을 조회(읽기)
2. `newQuantity = item.quantity + delta` 계산
3. `InventoryItems` 갱신(쓰기)
4. `InventoryLogs`에 기록(쓰기)

이 네 단계가 **`_db.transaction()`으로 감싸여 있지 않다**(코드 확인 결과 — `cancelOrder()`는 트랜잭션을 쓰는데 `adjustQuantity()`는 쓰지 않음, 두 메서드 간 비대칭이 이미 존재). 따라서:

- 두 결제(또는 두 직원의 수기 입력)가 같은 품목에 거의 동시에 `adjustQuantity()`를 호출하면, **둘 다 1단계에서 같은 `item.quantity`를 읽고**, 각자 독립적으로 새 값을 계산해 순서대로 덮어쓴다 — **고전적인 lost update**(나중에 쓴 값이 먼저 쓴 값의 효과를 지워버림).
- §3에서 확인했듯 **지금은 결제와 재고가 연동돼 있지 않아 이 위험이 실제로 발생하지 않는다**(재고 변경은 항상 한 직원이 화면에서 순차적으로 입력). 그러나 **향후 결제-재고 연동이 추가되면(다지점/다태블릿 환경에서 동시 결제가 흔해짐), 이 비원자성이 그대로 실제 매출-재고 불일치로 이어진다** — A-5(이력 보존)를 다루는 지금 이 비대칭을 인지해두는 것이 후속 위험을 줄인다.

## 6. `staffId`(A-1)와 재고 차감 주체 연결 구조

- `InventoryLog.staffId`는 nullable로 이미 존재하며, `adjustQuantity()`가 호출 시 받은 값을 그대로 저장한다.
- A-1(`PaymentRepository._staffIdOf()`)이 `OrderItem.staffId`에서 담당 스태프를 조달하는 것과 **같은 류의 패턴**이지만, **현재 재고 모듈에는 그 값을 자동으로 조달해주는 로직이 없다** — 14번 화면에서 직원이 직접 입력(또는 로그인 컨텍스트, 그러나 `A1_PREFLIGHT_REVIEW.md` §0에서 이미 확인했듯 로그인/세션 자체가 코드에 없음)해야 한다.
- **A-1과 A-5 사이에 직접적인 연결 고리는 없다** — `staffId`라는 개념을 같은 방식(nullable text, 검증 없음)으로 쓰고 있다는 유사성만 있을 뿐, 한쪽 모듈의 변경이 다른 쪽에 전파되는 구조가 아니다. F-INV-00 분리 원칙이 여기서도 그대로 유지된다.

## 종합 — 출력 형식

### 현재 구조로 충분한 부분
- 입고/사용/폐기 3가지 흐름은 `reason` enum 검증과 함께 `InventoryLog`에 자동 기록되고 있어, **"이력이 남는다"는 기본 요건 자체는 이미 충족**
- `deleteItem()` 테스트가 정상삭제 케이스를 검증하지 않아, **동작을 바꿔도 기존 테스트 회귀가 없음**(A-4와 동일한 안전한 출발점)
- F-INV-00(상품/결제와 완전 분리)이라는 원칙은 본 점검에서도 일관되게 유지되고 있어, **A-5 작업이 다른 모듈(A-1/A-2/A-4)에 영향을 주지 않는다**(역으로도 영향받지 않음)

### 부족한 부분
- **재고조사(棚卸)가 구조적으로 약함**: "조사 시점의 시스템수량/실사수량/차이"를 한 묶음으로 보존하는 메커니즘이 없음(`CashCount`와 비교해 구조화 수준이 낮음)
- **"현재 수량"과 "이력 합산"이 서로 다른 소스라 정합성 검증이 없음**: 둘이 어긋나도 감지할 방법이 없음(§2/§7)
- **`adjustQuantity()`가 트랜잭션으로 묶여 있지 않음**: `cancelOrder()`와의 비대칭, 동시성 문제의 직접 원인(§5)
- **`InventoryItem`에는 `Staff.accountStatus`처럼 재활용 가능한 "여유 필드"가 없음**: `A4_PREFLIGHT_REVIEW.md`가 채택한 "기존 컬럼에 새 값 추가" 전략을 A-5에 그대로 적용할 수 없다 — `name`/`category`/`unit`/`threshold`/`quantity` 중 의미를 훼손하지 않고 "비활성 상태"를 표현할 수 있는 필드가 없음(§"권장 설계 방향"에서 다룸)

### 확장 시 위험 요소
- **결제-재고 연동이 미래에 추가되는 순간, §5의 비원자성이 곧바로 실매출 손실로 연결됨**: 동시 결제가 잦은 영업시간대에 lost update가 발생하면, 실제로는 소진된 자재가 시스템상 남아있는 것으로 보여 재고부족을 못 감지하거나, 반대로 멀쩐히 있는 자재가 부족하다고 잘못 경고할 수 있음
- **세무조사 시 `InventoryLog`를 근거자료로 제출했는데 합계가 `quantity`와 다르면, 어느 쪽도 신뢰할 수 없는 상태가 됨**: A-5가 "이력은 안 지워진다"는 것만 보장하고 "이력이 정확하다"는 것은 보장하지 못하면, 감사 대응력이 기대만큼 올라가지 않음
- **품목 삭제 정책을 결정하지 않고 방치하면, A-4와 같은 종류의 고아데이터 문제가 재고 쪽에서도 반복됨**(이미 `DESIGN_REVIEW.md` §3에서 식별된 위험의 재확인)

### 권장 설계 방향 (코드/SQL 생성 없이, 방향만)

1. **`deleteItem()`의 정책을 "상태 전환"이 아니라 "이력이 있으면 삭제 자체를 거부"하는 방향으로 잡는 것을 권장한다.** `A4_PREFLIGHT_REVIEW.md`처럼 기존 컬럼을 재활용해 상태값을 추가하는 방식은, `InventoryItem`에 그럴 여유 필드가 없어 **그대로 적용할 수 없다.** 대신 `deleteItem()` 호출 시 "이 품목을 참조하는 `InventoryLog`가 1건 이상 있으면 거부"하는 검증을 추가하면, **컬럼 추가 없이도 이력이 있는 품목은 영구 보존된다**(이력이 전혀 없는, 등록 직후 실수로 만든 품목만 삭제 가능 — 이 경우는 보존할 이력 자체가 없으므로 문제 없음).
2. **재고조사(棚卸)는 본 A-5 범위에 포함하지 않고 별도 항목으로 분리할 것을 권장한다.** 구조화하려면 신규 컬럼/엔티티가 필요해질 가능성이 높아(`CashCount`급 구조화), 지금은 "조사 시 메모에 조사일자/실사수량을 남기는 운영 절차"로 1차 완화하고, 구조화는 `BUSINESS_RULES_TO_DECIDE.md`/`SCREENS_ERD_TABLES.md` §4의 제안 목록 결정 이후로 미루는 것이 일관성 있다.
3. **`adjustQuantity()`를 트랜잭션으로 감싸는 것을 향후 과제로 명시할 것을 권장한다.** 지금 당장 결제 연동이 없어 위험이 표면화되지 않았지만, A-5 구현 시점에 "현재는 비원자적"이라는 사실을 코드 주석/문서로 명확히 남겨, 향후 결제-재고 연동을 검토할 때 이 부분을 반드시 같이 재검토하도록 연결고리를 남겨둔다.
4. **"현재 수량"과 "이력 합산" 간 정합성 검증 로직(읍 read-only 점검 함수)을 향후 별도로 추가할 가치가 있다는 점만 기록하고, 본 A-5 1차 범위에는 포함하지 않는다.** 이력 보존(삭제 방지)과 정합성 검증은 서로 다른 문제이며, 전자를 먼저 해결해도 후자 없이는 "보존된 이력이 맞는지"를 확신할 수 없다는 한계를 인지한 채로 1단계를 진행하는 것이 현실적이다.
