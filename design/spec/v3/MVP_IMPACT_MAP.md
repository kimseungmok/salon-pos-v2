# MVP 필수 5건 — 영향범위 및 구현순서 설계 문서

> **기준 문서**: `PRIORITIZATION.md` §A(MVP 출시 전 반드시 필요한 항목)
> **목적**: 5건 각각이 어떤 화면·테이블·Repository에 영향을 주는지 매핑하고, 의존관계를 고려한 구현 순서를 정한다.
> **범위 제한**: 설계 문서만 작성. 코드/SQL 생성 없음, 테이블 구조도 변경하지 않는다(영향받는 테이블을 "식별"만 함).
> 작성일: 2026-06-23

---

## A-1. 방문완료 시 `recordVisit()` 호출 연결

### 영향받는 화면
| 화면 | 영향 내용 |
|---|---|
| 09_customer_list ✅ | 그룹탭(初回来店/常連/休眠ぎみ) 분류 결과가 비로소 실데이터로 채워짐 — 직접적 수혜 화면 |
| 10_customer_detail 📄 | 방문이력 탭이 표시할 데이터의 원천 — 이 항목이 해결돼야 화면 구현 의미가 생김 |
| 02_pos_order / 03_pos_payment / 04_pos_complete | 결제완료를 "방문완료" 트리거로 쓸 경우의 호출 시작점 후보 |
| 06_booking_calendar / 07_booking_form | 예약완료를 "방문완료" 트리거로 쓸 경우의 호출 시작점 후보(A-2와 연결) |
| 08_waiting_list ✅ | 워크인(예약 없이 온 고객)이 시술 종료 후 방문완료 처리되는 경로 — 이 경로도 누락하면 워크인 고객은 영원히 이력이 안 쌓임 |

### 영향받는 테이블
| 테이블 | 영향 내용 |
|---|---|
| VisitRecord | 쓰기 대상(새 행 insert) — 본 항목의 직접 목적 |
| Customer | 읽기(고객 존재 확인), 그룹분류 산출 시 참조 |
| Booking | 읽기(트리거 소스 — A-2와 동시에 고려 필요) |
| Order / Payment | 읽기(트리거 소스 — 결제완료 시점을 쓸 경우) |

### 영향받는 Repository
| Repository | 영향 내용 |
|---|---|
| CustomerRepository | `recordVisit()` 보유 — 변경 없이 "호출되는 쪽"이 추가됨 |
| BookingRepository | 예약완료 트리거를 채택할 경우 호출 시작점 후보(A-2 처리와 동일 위치일 가능성 높음) |
| PaymentRepository | 결제완료 트리거를 채택할 경우 호출 시작점 후보 — 워크인(예약 없는) 고객의 유일한 트리거 경로이므로 **어느 한쪽만 선택해도 안 되고, 두 경로 모두 고려 필요** |

### 예상 구현 순서
**3번째** — A-2(방문완료 상태 정의)가 먼저 정리되어야 "예약경로"의 트리거 시점이 명확해짐. 다만 "결제경로"(워크인) 트리거는 A-2와 무관하게 독립적으로 먼저 설계할 수 있어, 실제로는 A-2와 병행 검토가 가능.

### 예상 난이도
**하** — 새 테이블/메서드 불필요. 기존 `CustomerRepository.recordVisit()`을 누가, 언제 호출하느냐만 정하면 됨. 다만 "예약경로"와 "결제경로" 두 트리거가 중복 호출되지 않도록(한 번의 방문에 VisitRecord가 2건 쌓이지 않도록) 호출 시점을 한 곳으로 일관되게 정리하는 설계 판단이 필요해, 순수 코드량보다 의사결정 비중이 큼.

## A-2. 예약 방문완료 상태 전환

### 영향받는 화면
| 화면 | 영향 내용 |
|---|---|
| 06_booking_calendar / 06_booking_calendar_b 📄 | 완료된 예약을 캘린더에서 구분 표시(노쇼 칩과 동일한 위치에 완료 칩 추가 여지) |
| 07_booking_form 📄 | 예약 상세 팝업에서 "来店完了" 처리 버튼이 필요해짐 |
| 08_waiting_list ✅ | 기존 `callWaiting()`/`cancelWaiting()` 패턴(상태 전환 함수)이 참고 모델이 될 수 있음 |

### 영향받는 테이블
| 테이블 | 영향 내용 |
|---|---|
| Booking | `status` 컬럼을 `'completed'`로 갱신(컬럼은 이미 존재, 갱신 로직만 없음) |

### 영향받는 Repository
| Repository | 영향 내용 |
|---|---|
| BookingRepository | `cancelBooking()`과 대칭되는 `completeBooking()` 류의 메서드가 필요한 지점 — 기존 `cancelBooking()`의 상태검증 패턴(이미 cancelled/noshow인 경우 차단) 재사용 가능 |

### 예상 구현 순서
**1번째** — 다른 어떤 항목에도 의존하지 않는 독립 항목이며, A-1의 "예약경로" 트리거가 성립하기 위한 전제조건이라 가장 먼저 정리하는 것이 합리적.

### 예상 난이도
**하** — `cancelBooking()`과 거의 동일한 구조(상태 검증 + 단순 업데이트)로 처리 가능, 새로운 패턴을 고안할 필요가 없음.

## A-3. 예약 변경(`updateBooking`) — 시간/담당자 재배정 포함

### 영향받는 화면
| 화면 | 영향 내용 |
|---|---|
| 06_booking_calendar / 06_booking_calendar_b 📄 | 예약 블록을 드래그/클릭해 시간을 바꾸는 흐름의 데이터 처리 대상 |
| 07_booking_form 📄 | "수정" 모드 진입 시 기존 값을 불러와 재검증하는 폼 동작 필요 |

### 영향받는 테이블
| 테이블 | 영향 내용 |
|---|---|
| Booking | 거의 전 컬럼(`staffId`/`productIdsCsv`/`startAt`/`endAt`/`depositAmount` 등)이 갱신 대상 |
| Shift | 담당자 재배정 시 가용여부 재검증을 위한 읽기 대상 |
| Booking(자기참조 비교) | 충돌검사 시 "수정 대상 예약 자신"을 충돌 후보에서 제외해야 함(현재 `createBooking()`의 `_assertStaffAvailable()`은 신규 생성 전제라 이 예외처리가 없음) |

### 영향받는 Repository
| Repository | 영향 내용 |
|---|---|
| BookingRepository | `updateBooking()` 신설 대상, `_assertStaffAvailable()`을 "자기 자신 제외" 옵션과 함께 재사용해야 함 |
| StaffRepository | `isOnShift()` 호출 경로(기존 그대로 재사용, 변경 불필요) |

### 예상 구현 순서
**5번째(마지막)** — A-1/A-2(방문완료 흐름)와 직접적인 의존관계는 없지만, `_assertStaffAvailable()` 충돌검사 로직을 "생성"과 "수정" 양쪽에서 공유해야 하므로, 다른 항목들이 먼저 안정화된 뒤 booking 모듈을 한 번에 정리하는 편이 재작업을 줄임. 5건 중 영향범위(컬럼 수)가 가장 넓어 마지막으로 배치.

### 예상 난이도
**중** — 충돌검사에서 "자기 자신 제외" 예외처리, 예약금이 이미 수령된 상태에서 금액/시간이 바뀔 때의 처리 정책(전액유지/재계산) 등 단순 CRUD보다 분기가 많음.

## A-4. 직원 재직/퇴사 상태 — `removeStaff()` 하드 삭제 제거

### 영향받는 화면
| 화면 | 영향 내용 |
|---|---|
| 11_staff_list 📄 | 재직/퇴사 상태를 구분 표시할 컬럼 필요(현재 화면 정의에 재직상태 개념 자체가 없음) |
| 12_staff_detail 📄 | 퇴사 처리 버튼의 동작 대상 |
| 33_staff_accounts ✅ | 현재 구현된 화면 — `removeStaff()`를 직접 호출하는 실제 호출처(가장 먼저 동작이 바뀌는 화면) |

### 영향받는 테이블
| 테이블 | 영향 내용 |
|---|---|
| Staff | 삭제 대신 상태값을 갱신하는 방식으로 동작 변경(컬럼 자체는 식별만, 추가는 본 문서 범위 밖) |
| VisitRecord / OrderItem / Booking / InventoryLog / CashCount | 모두 `staffId`를 텍스트로 참조 중 — 직접 갱신 대상은 아니지만, "퇴사 직원도 과거 기록에서 이름이 계속 보이는지"를 검증할 때 함께 조회되는 테이블들 |

### 영향받는 Repository
| Repository | 영향 내용 |
|---|---|
| StaffRepository | `removeStaff()`의 동작을 하드 삭제에서 상태 전환으로 바꾸는 핵심 변경 지점 |
| BookingRepository | `_assertStaffAvailable()`이 "퇴사한 직원에게는 새 예약을 못 잡게" 막아야 하는지 여부를 함께 검토해야 함(퇴사 상태 도입의 부수 효과) |

### 예상 구현 순서
**2번째** — 다른 4건과 의존관계가 없는 독립 항목이며, "삭제하면 영구 손상"이라는 리스크 특성상 더 미룰수록 실제 손상이 발생할 가능성이 커짐. A-2 다음으로 빠르게 처리하는 것이 안전.

### 예상 난이도
**중** — 동작 자체(삭제→상태전환)는 단순하지만, 기존에 이미 하드 삭제된 데이터는 되돌릴 수 없고, "퇴사 직원을 목록에서 어떻게 보여줄지"(완전히 숨김 vs 흐리게 표시) 등 화면 정책 결정이 함께 필요.

## A-5. 재고품목 삭제 시 이력 동반삭제(cascade) 방지

### 영향받는 화면
| 화면 | 영향 내용 |
|---|---|
| 14_inventory_list ✅ | 품목 삭제 버튼의 실제 동작 대상(현재 구현된 화면) |
| 15_inventory_history 📄 | 삭제된 품목의 이력이 조회 목록에서 사라지는 현상과 직결되는 화면 |

### 영향받는 테이블
| 테이블 | 영향 내용 |
|---|---|
| InventoryItem | 삭제 대신 상태 전환으로 동작이 바뀌는 대상(컬럼 추가는 본 문서 범위 밖, 식별만) |
| InventoryLog | 현재 `KeyAction.cascade`로 동반 삭제되는 대상 — 본 항목의 핵심 보호 대상 |

### 영향받는 Repository
| Repository | 영향 내용 |
|---|---|
| InventoryRepository | `deleteItem()`의 동작을 하드 삭제에서 상태 전환(또는 이력 보존 정책)으로 바꾸는 핵심 변경 지점 |

### 예상 구현 순서
**4번째** — A-4와 성격이 유사(하드 삭제→상태전환)하지만, 재고 품목 삭제는 직원 퇴사보다 발생 빈도가 낮아 우선순위를 한 단계 낮게 배치. 다만 A-4와 같은 패턴을 쓸 가능성이 높아 **A-4 작업 직후 이어서 진행하면 설계 일관성을 유지하기 쉬움**.

### 예상 난이도
**중** — A-4와 동일하게 "동작은 단순하나 화면 정책 결정이 필요"한 유형. 다만 재고는 `PrepaidPassMenu.status(active/disabled)`라는 이미 검증된 패턴이 코드베이스 안에 있어, A-4보다 참고할 선례가 명확한 편.

## 종합 — 구현 순서 요약

| 순서 | 항목 | 핵심 이유 |
|---|---|---|
| 1 | A-2 예약 방문완료 상태전환 | 독립 항목 + A-1의 전제조건 |
| 2 | A-4 직원 재직/퇴사 상태 | 독립 항목 + 데이터 손상 리스크가 시간이 갈수록 커짐 |
| 3 | A-1 recordVisit() 호출 연결 | A-2 완료 후 예약경로 트리거가 명확해짐(결제경로는 병행 가능) |
| 4 | A-5 재고품목 이력보존 | A-4와 동일 패턴(하드삭제→상태전환), 직후 진행이 설계 일관성에 유리 |
| 5 | A-3 예약변경(updateBooking) | 영향범위가 가장 넓고 분기가 많아, 다른 항목이 안정화된 뒤 마지막에 정리 |

**난이도 분포**: 하(A-1, A-2) → 중(A-3, A-4, A-5). 영향도가 가장 큰 A-1/A-2가 동시에 난이도도 가장 낮으므로, 위 순서대로 진행하면 초반에 빠르게 핵심 리스크를 해소하고 후반에 상대적으로 복잡한 작업을 배치하는 흐름이 된다.
