# `Staff.accountStatus` 시스템 전체 공통 규칙 확정

> **기준 문서**: `A4_PREFLIGHT_REVIEW.md`, `A4_A5_INTEGRATED_REVIEW.md`, `A4_A5_PAYMENT_FLOW_VERIFICATION.md`
> **목적**: 지금까지 분석/검증 단계에 머물러 있던 `accountStatus`의 의미를 **프로젝트 전체에서 동일하게 해석되는 단일 규격**으로 확정한다 — 이후 어떤 화면/Repository를 구현하든 이 문서 하나만 참조하면 된다.
> **범위 제한**: 설계 문서만 작성. 코드/SQL 생성·수정, 테이블/컬럼 변경 없음. `accountStatus`는 이미 `nullable TEXT`로 존재하는 컬럼이며, 본 문서는 그 컬럼에 **어떤 문자열이 허용되고 각각 무엇을 뜻하는지**만 확정한다.
> 작성일: 2026-06-24

---

## 1. 상태 정의 확정

### 1-1. 허용 값 목록 (전체 4종, 이 외에는 절대 사용하지 않음)

| 값 | 한국어 의미 | 시스템 관점 정의 |
|---|---|---|
| `null` | (미사용 — 아래 §4-2 참조) | 컬럼이 `nullable`이라는 스키마적 사실의 잔재일 뿐, **앱 코드가 명시적으로 생성하는 값이 아니다.** 현재 `inviteStaff()`는 항상 `'待機中'`으로 시작하므로 실제 데이터 경로에서 `null`이 만들어지지 않는다. 화면의 `?? '未招待'` 같은 방어적 처리는 유지하되, 이를 "초대 자체를 안 보낸 상태"로 새로 쓰려는 시도는 하지 않는다(혼란 방지, §4-2) |
| `'待機中'` | 초대됨, 매출장부 연결 대기 | F-STAFF-01 초대 흐름 진행 중. **이미 매장에서 근무 중일 수 있다** — 이 값은 "근무 여부"가 아니라 "외부 매출장부 시스템과의 계정 연결 상태"만을 뜻함 |
| `'連結済み'` | 정식 연결완료 | 초대 수락 완료, 매출장부 시스템과 연결됨. 재직 중인 상태의 기본값 |
| `'退職済み'` | 퇴사처리됨 | A-4가 도입하는 신규 값. `'連結済み'` 상태였던 직원이 퇴사할 때만 전환된다(§3-2) — **그 외 상태에서 이 값으로 전환되는 경로는 없다** |

### 1-2. 각 값의 의미를 명확히 구분하는 핵심 원칙
**`accountStatus`는 "근무 여부"가 아니라 "계정 연결 진행상태 + 퇴사 여부"라는 단일 축으로만 해석한다.** `'待機中'`인 사람도 실제로는 매장에서 일하고 있을 수 있다는 점이 가장 흔히 오해되는 지점이다 — 이 값은 "이 사람이 퇴사했는가"만 가른다고 생각하면 전체 시스템 영향범위(§2)가 단순해진다.

## 2. 시스템 영향 범위

### 2-1. Booking — 상태별 예약 가능 여부
**단일 규칙**: `accountStatus != '退職済み'`이면 신규 예약 배정 가능, `'退職済み'`면 불가능.

| 상태 | 신규 예약 배정 |
|---|---|
| `null` | ✅ 가능(이론상 도달 안 하는 값이지만, 도달해도 차단 대상 아님) |
| `'待機中'` | ✅ 가능 — 매출장부 미연결과 무관하게 실제 근무자일 수 있으므로 |
| `'連結済み'` | ✅ 가능 |
| `'退職済み'` | ❌ 불가능 — `BookingRepository._assertStaffAvailable()`에 검증 추가 대상(`A4_A5_PAYMENT_FLOW_VERIFICATION.md`가 "연결해야 하는 영역"으로 이미 확정) |

기존에 그 직원에게 배정된 예약(과거/미래 모두)은 퇴사 후에도 **그대로 유지**된다 — 이 규칙은 "신규 배정"에만 적용되고, 기존 `Booking.staffId`를 되돌리거나 취소시키지 않는다.

### 2-2. Payment — 상태와 무관하게 결제 가능 여부
**`accountStatus`를 전혀 조회하지 않는다.** `PaymentRepository.pay()`는 `OrderItem.staffId`를 그대로 정산하며, 그 직원이 현재 어떤 상태든(퇴사했어도) 결제는 항상 성립한다 — `A4_A5_PAYMENT_FLOW_VERIFICATION.md` §1-1에서 이미 확정한 "시간차 오류 방지" 원칙 그대로다. **이 규칙에 예외는 없다.**

### 2-3. Visit — 과거 기록 유지 및 조회 정책
`VisitRecord.staffId`는 기록 시점의 값을 영구히 보존하며, **조회 시 `accountStatus`로 필터링하지 않는다.** 퇴사한 직원의 과거 방문기록도 항상 그 직원 이름으로 조회 가능해야 한다(`DESIGN_REVIEW.md` §5가 이미 정한 "퇴사직원의 과거 매출 기여도 인정" 원칙).

### 2-4. Staff 선택 UI — 표시/비표시 기준
화면 목적에 따라 기준이 다르므로, **목적별로 명확히 구분해 고정**한다.

| 화면 목적 | 표시 기준 |
|---|---|
| 33번(직원 초대/계정 관리) | **전체 표시** — `'退職済み'`도 보여야 함(재초대/관리 대상이므로 숨기면 안 됨) |
| 06/07(예약 시 담당자 선택, 향후 구현) | `accountStatus != '退職済み'`만 표시 — §2-1과 동일 규칙 |
| 11번(직원목록, 향후 구현) | 기본 탭은 `accountStatus != '退職済み'`만, 별도 탭/필터로 퇴사자 조회 가능하게 — 하드 숨김이 아니라 기본값만 활성직원으로 |

## 3. 데이터 무결성 규칙

### 3-1. 상태 변경 시 기존 데이터(과거 예약/매출) 영향 여부
**영향 없음 — 연쇄 갱신을 절대 만들지 않는다.** `accountStatus`를 갱신하는 연산은 `Staff` 테이블의 해당 행 **하나만** 바꾼다. `Booking.staffId`/`OrderItem.staffId`/`VisitRecord.staffId`/`InventoryLog.staffId`/`CashCount.confirmedBy` 중 어느 것도 이 갱신에 의해 같이 바뀌지 않는다(`A4_A5_PAYMENT_FLOW_VERIFICATION.md` §1-3에서 이미 확정).

### 3-2. 삭제 대신 상태 변경을 사용하는지 여부 — 이원화 규칙
`removeStaff()`는 **현재 `accountStatus` 값에 따라 동작이 갈린다**(`A4_A5_INTEGRATED_REVIEW.md` §1-1에서 도출된 규칙을 본 문서가 공식 확정):

| 호출 시점의 `accountStatus` | `removeStaff()` 동작 |
|---|---|
| `null` 또는 `'待機中'` | **하드 삭제 유지** — 정식 연결 전이므로 보존할 이력이 구조적으로 없음(예외적으로 이 상태에서 `Shift`/`Booking`이 미리 생성된 극단적 케이스는 빈도가 매우 낮아 1차 정책에서 무시) |
| `'連結済み'` | **상태 전환**(`accountStatus = '退職済み'`) — 하드 삭제 금지 |
| `'退職済み'`(이미 퇴사 처리된 사람을 다시 삭제 시도) | 멱등 처리 — 추가 동작 없이 그대로 두거나 `BusinessRuleException`(중복 처리), 본 문서는 구체 구현을 정하지 않고 "다시 하드 삭제로 빠지면 안 된다"는 점만 고정 |

**재입사(복귀) 처리**: `'退職済み'`였던 직원이 다시 근무하게 되면, **같은 `Staff.id`를 재사용해 `accountStatus`를 `'連結済み'`로 되돌리는 것을 허용한다.** 새 ID로 별도 생성하지 않는다 — 매출/예약 이력의 연속성(같은 사람의 과거 실적을 한 ID 아래 계속 누적)이 새 ID 분리보다 더 중요하다고 판단. 퇴사기간 중 발생한 신규 예약/매출은 §2-1 규칙(퇴사 중엔 신규 배정 불가)에 의해 원천적으로 없으므로, 복귀 후 이력이 섞이는 문제는 발생하지 않는다.

## 4. 혼재 방지 규칙

### 4-1. 잘못된 문자열 값 방지 기준
`accountStatus`는 여전히 자유 텍스트 컬럼이라(컬럼 변경 금지 조건상 `CHECK` 제약을 새로 걸 수 없음), **DB 레벨에서 잘못된 값 입력을 막을 방법이 없다.** 따라서:
- **애플리케이션 레벨에서만 강제 가능하다는 한계를 명확히 인지한다** — `StaffRepository`가 이 4종 값만을 상수로 다루고, 그 외 문자열을 쓰는 코드 경로를 만들지 않는 것이 유일한 방어선이다.
- 화면이든 Repository든 **"이 4개 값 목록"을 한 곳(`StaffRepository` 또는 그에 준하는 단일 위치)에서만 정의**하고, 다른 모듈이 별도로 같은 의미의 다른 문자열(예: `'active'`/`'inactive'` 같은 영문 별칭)을 만들어 쓰지 않는다.

### 4-2. 기본값 처리 방식
- 신규 직원은 `inviteStaff()`를 통해 항상 `'待機中'`으로 시작한다(기존 동작 그대로, 변경 없음).
- `null`은 "기본값"으로 쓰지 않는다 — §1-1에서 확정한 대로, 앱 코드가 의도적으로 생성하는 값이 아니다. **신규 직원 생성 경로에 "초대 없이 직원만 등록"하는 기능이 향후 추가되더라도, 그 경우의 기본값은 `null`이 아니라 `'待機中'`(또는 그에 준하는 명시적 값)으로 고정**하는 것을 원칙으로 한다 — `null`이 의미를 갖는 상태로 격상되지 않도록 지금 미리 막아둔다.

## 출력 형식

### `accountStatus` 최종 규격 1안

```
허용값: null(미사용, 방어적 처리용) | '待機中' | '連結済み' | '退職済み'
전이 규칙: null/'待機中' --(초대 수락)--> '連結済み'
           '連結済み' --(퇴사처리)--> '退職済み'
           '退職済み' --(복귀, 같은 ID 재사용)--> '連結済み'
신규배정 가능 조건: accountStatus != '退職済み'
결제/방문기록 조회: accountStatus 완전 무관(항상 그대로 조회/처리)
삭제 정책: null/'待機中' = 하드삭제 유지, '連結済み' 이상 = 상태전환만
```

### 절대 금지되는 상태 정의 방식
- **`accountStatus`에 "근무 여부"가 아닌 다른 의미(직급, 권한, 출근상태 등)를 같이 싣는 것** — F-STAFF-00 원칙(권한/PIN은 본 앱 책임 밖)과 충돌하며, 이 필드의 단일 축 해석(§1-2)을 깨뜨림
- **이 문서에 정의된 4종 외의 새 값을 사전 협의 없이 추가하는 것**(예: `'休職中'`, `'長期休暇'` 등) — 추가가 필요해지면 본 문서를 갱신하는 절차를 먼저 거쳐야 함
- **화면/모듈마다 다른 판정 기준식을 쓰는 것**(예: 어떤 화면은 `== '連結済み'`로 "재직"을 판단하고 다른 화면은 `!= '退職済み'`로 판단하는 등 — 의미상 다른 결과를 낼 수 있는 서로 다른 식을 혼용하면 안 됨, §2-1/§2-4의 단일 식으로 통일)
- **`null`을 "재직 중"이나 "퇴사함" 등 의미있는 상태로 새로 해석하려는 시도** — §4-2에서 이미 "미사용 값"으로 고정함
- **DB 레벨 `CHECK` 제약이나 별도 enum 컬럼으로 강제하려는 시도** — 컬럼 변경 금지 조건과 정면 충돌, 시도 자체를 하지 않는다

### 시스템 전체 영향 요약
| 모듈 | 영향 |
|---|---|
| Booking | 신규 배정 시 `!= '退職済み'` 검증 추가 필요(유일한 실제 코드 변경 지점) |
| Payment | 영향 없음(상태 비조회 원칙 유지) |
| Visit(Customer) | 영향 없음(과거기록 비필터링 원칙 유지) |
| Staff 화면(11/12/33) | 화면별로 다른 표시기준(§2-4) — 구현 시 참조할 단일 규격 확보 |
| Shift | 영향 없음(상태와 무관하게 그대로 유지) |

### A-5(Inventory)와 충돌 여부
**충돌 없음.** `InventoryLog.staffId`는 §2-3(Visit)과 동일한 성격(과거 기록)이므로, 같은 원칙(상태 무관 조회·비필터링)을 그대로 적용하면 충돌이 발생하지 않는다. 향후 `InventoryRepository`에 "재고 처리 권한자 검증" 같은 기능이 추가되더라도, 그 시점에는 §2-1(Booking)과 같은 분류(신규 작업 배정 시점)에 속하게 되므로 **같은 단일 규칙(`!= '退職済み'`)을 그대로 재사용**하면 된다 — 새로운 규칙을 따로 만들 필요가 없다는 점에서, 본 규격은 Inventory 확장 시에도 일관되게 적용 가능하다.
