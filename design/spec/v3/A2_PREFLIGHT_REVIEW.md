## 5. 구현 순서 (A-2 내부 세부 단계)

1. `cancelBooking()`의 상태검증 분기(이미 `cancelled`/`noshow`면 차단)를 참고해, **`completeBooking()`이 차단해야 할 기존 상태 목록을 먼저 확정**(§6에서 다룰 "이미 cancelled/noshow인 예약을 완료 처리하려는 시도"가 핵심 분기)
2. `BookingRepository`에 `completeBooking()` 추가(상태를 `'completed'`로 전환)
3. `booking_repository_test.dart`에 대칭 테스트 그룹 추가(정상 완료 1건, 중복 완료 차단 1건, 존재하지 않는 예약 1건, cancelled/noshow 예약을 완료처리하려는 시도 차단 1건 — 최소 4케이스)
4. (A-2 범위 밖, 차후) 06/07 화면 구현 시점에 `completeBooking()`을 실제로 호출하는 UI 버튼 연결
5. (A-2 범위 밖, 차후) A-1 단계에서 `completeBooking()` 성공 직후 `CustomerRepository.recordVisit()`을 호출하는 연결 코드 추가

**본 문서가 다루는 범위는 1~3단계까지**이며, 4~5단계는 각각 06/07 화면 구현·A-1 항목의 책임으로 분리해 둔다(혼선 방지).

## 6. 예상 리스크

| 리스크 | 설명 |
|---|---|
| **"화면 없는 기능"이 먼저 들어가는 구조적 어색함** | 06/07이 미구현이라 `completeBooking()`을 호출할 UI가 당장 없다. 구현 후 한동안 "테스트로만 존재하는 메서드"가 되어, 실제로 동작 확인이 늦어질 수 있음 — A-1과 너무 떨어진 시점에 구현하면 "왜 만들었는지" 맥락이 흐려질 위험 |
| **A-1과의 호출 순서를 잘못 정하면 이중 적재 위험** | `completeBooking()`이 내부적으로 `recordVisit()`을 같이 호출하도록 설계할지, 아니면 완전히 분리해서 호출자(화면 또는 PaymentRepository)가 둘을 순서대로 호출하도록 할지 — 이 경계를 A-2 구현 시점에 미리 정해두지 않으면, A-1 구현 시점에 동일한 방문에 대해 `VisitRecord`가 중복 생성될 위험이 있음 |
| **결제경로(워크인)와의 정합성 미정** | `MVP_IMPACT_MAP.md`가 이미 지적했듯, 방문완료 트리거는 "예약경로"(`completeBooking()`)와 "결제경로"(워크인, Payment 완료)가 별도로 존재한다. A-2는 예약경로만 다루므로, **예약이 있는 고객의 방문완료가 "예약완료 처리"와 "결제완료" 중 어느 시점에 최종 확정되는지**(둘 다 처리해야 하는지, 결제완료가 예약완료를 자동으로 트리거해야 하는지)에 대한 결론이 아직 없음 — A-2 구현 전에 이 순서 관계만이라도 정해두는 것이 바람직 |

## 7. 예외 케이스 (구현 시 반드시 분기 처리해야 할 시나리오)

| # | 시나리오 | 처리 방향(권장, 미확정) |
|---|---|---|
| 1 | 이미 `'completed'`인 예약을 다시 완료 처리하려는 시도 | `cancelBooking()`의 기존 패턴과 동일하게 `BusinessRuleException`으로 차단 — 멱등성 보장 |
| 2 | `'cancelled'` 또는 `'noshow'`인 예약을 완료 처리하려는 시도 | 차단 필요. 노쇼로 이미 처리된 예약이 나중에 "사실 왔었다"로 뒤집히는 케이스는 본 메서드가 아니라 별도 정정 절차로 다뤄야 함(본 항목 범위 밖이라는 점을 명시) |
| 3 | 존재하지 않는 `bookingId`로 완료 처리 시도 | `NotFoundException` — `cancelBooking()`과 동일 패턴 |
| 4 | 시작시각(`startAt`)이 아직 도래하지 않은 예약을 완료 처리하려는 시도(미래 예약을 실수로 완료 처리) | 현재 `cancelBooking()`에는 이런 시간 검증이 없음 — `completeBooking()`에 추가할지는 **결정 필요**(운영상 실수 방지 차원에서는 추가가 바람직하나, "현장에서 시술이 예정보다 빨리 끝나 미리 완료처리"하는 정당한 케이스와 충돌할 수 있어 신중한 판단 필요) |
| 5 | 같은 예약을 완료 처리와 취소 처리가 동시에 시도되는 동시성 케이스(드물지만 분할 결제 화면과 캘린더 화면을 동시에 조작하는 경우) | 둘 다 같은 사전조회→검증→갱신 패턴이라 Drift의 단일 트랜잭션 내 갱신으로 원자성은 보장되나, "먼저 도착한 요청이 이긴다"는 동작이 사용자에게 어떻게 안내되는지는 화면단(06/07 구현 시점)의 책임으로 분리 |
| 6 | 예약금(`depositReceived=true`)이 있는 예약을 완료 처리하는 경우 | `cancelBooking()`은 `depositRefunded` 플래그를 함께 갱신하지만, **완료 처리는 예약금 정산과 무관**해야 함(완료=정상 방문이므로 예약금은 결제에 그대로 사용되는 것이 정상 흐름) — `completeBooking()`이 `depositRefunded`/`depositReceived` 필드를 건드리지 않도록 명확히 범위를 한정해야 함 |

## 종합 결론

A-2는 **테이블/Repository 영향범위가 매우 좁고(`Booking.status` 단일 컬럼, `BookingRepository` 단일 클래스) 난이도도 낮지만, 화면이 아직 없어 "만들어도 당장 쓰이지 않는" 상태로 먼저 들어가는 항목**이다. 구현 자체보다 **A-1(결제경로와의 트리거 순서)과의 경계를 미리 정하는 의사결정**이 실질적인 선행 작업이며, 이 결정 없이 §6 두 번째 리스크(이중 적재)를 안고 구현을 시작하면 A-1 단계에서 재작업이 발생할 가능성이 높다. §7의 예외케이스 중 1~3은 기존 `cancelBooking()` 패턴을 그대로 따르면 해결되지만, 4(미래예약 완료처리 차단 여부)와 6(예약금 필드 비침범)은 구현 착수 전 명확한 결정이 필요하다.
