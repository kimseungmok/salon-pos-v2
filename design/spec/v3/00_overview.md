# salon-pos v3 정의서 — 개요

> **목적**: 이 문서 세트(v3)만 보고 일본 살롱 POS를 처음부터 구현해도 토스플레이스(TossPlace) 100% 벤치마킹 기준에서 벗어나지 않는 품질로 만들 수 있도록, 화면이 아니라 **자료조사(`../toss_benchmarking.md`) 기반의 정의서**로 다시 작성한다.
> **베이스 자료**: `design/spec/toss_benchmarking.md`(토스 실제 화면 검증 기록) + `design/mockups/v2/ja_toss100/`(기존 구현 목업, 1차 참고용이며 정의서가 우선)
> **작성 시작**: 2026-06-23

---

## 1. 문서 구성 원칙

영역(고객관리, 선불권 등) 하나당 항상 3종 문서를 세트로 만든다.

| 문서 | 파일명 | 내용 |
|---|---|---|
| 기능 정의서 (Feature Spec) | `feature_spec.md` | 기능이 "무엇을, 왜, 어떤 규칙으로" 동작하는지. 비즈니스 규칙·예외처리·제약조건 중심. 화면 레이아웃은 다루지 않음. |
| 화면 정의서 (Screen Spec) | `screen_spec.md` | 화면 번호·Zone 구조·UI 요소·상태분기·인터랙션 흐름. "어떻게 보이고 어떻게 동작하는지"를 화면 단위로. |
| 데이터 정의서 (Data Spec) | `data_spec.md` | 엔티티·필드·관계·산출 로직(pseudo-code). 기능정의서의 규칙을 실제 저장 구조로 변환. |

세 문서는 서로를 참조한다 — 기능정의서의 규칙 ID(예: `F-CUST-01`)를 화면정의서·데이터정의서에서도 그대로 인용해 추적 가능하게 한다.

## 3. 현지화 원칙 (필수)

`[[project_salon_pos_v2_principles]]` 메모리 참조 — 모든 화면 정의서의 UI 라벨은 **일본어 원문**으로 표기하고, 필요시 한국어 괄호 설명을 보조로 붙인다. 화폐는 ¥, 날짜는 年月日, 단위는 件/名/分/種을 사용한다. 본 문서(기능/데이터 정의서)는 작업 편의상 한국어 위주로 작성하되, 사용자에게 노출되는 모든 문자열(라벨/버튼/안내문구)은 일본어로 명시한다.

## 3-5. A-2 구현 전 최종 점검

`A2_PREFLIGHT_REVIEW.md` — A-2(예약 방문완료 상태전환) 착수 전 점검. 영향범위는 `Booking.status` 단일 컬럼·`BookingRepository` 단일 클래스로 매우 좁음. **선행 의사결정 필요**: A-1(결제경로 트리거)과의 호출 순서를 먼저 정하지 않으면 VisitRecord 이중 적재 위험. 예외케이스 6건 중 "미래예약 완료처리 차단 여부"·"예약금 필드 비침범"은 구현 전 결정 필요. 검토만 수행, 코드/컬럼 추가 없음(2026-06-23).

## 3-6. A-1×A-2 업무 경계 정의

`A1_A2_BOUNDARY.md` — 두 항목이 한 트리거(결제완료)에 종속되도록 경계 확정. **방문확정 = `Order.status→'completed'` 단일 트리거**(예약 유무 무관), `recordVisit()` 호출책임은 `PaymentRepository.pay()` 단일 지점으로 한정(BookingRepository/화면이 직접 호출 금지). `completeBooking()`은 bookingId가 전달된 경우에만 같은 트리거에서 부수 호출. 경계 밖으로 남긴 케이스: 0원 주문(전액할인)이 영원히 완결 안 되는 문제, 환불 시 VisitRecord 처리방식. 설계 경계만 정의, 코드 없음(2026-06-23).

## 3-7. 구현 착수 전 결정 필요한 업무 규칙 모음

`BUSINESS_RULES_TO_DECIDE.md` — 예약/고객/결제/재고/직원 5영역에서 DB설계·데이터적재에 영향을 주는 업무규칙을 즉시결정/구현중결정/출시전결정 3단계로 분류. **즉시결정 7건**: 방문확정 시점(예약완료 vs 결제완료), 0원주문 방문인정 여부, 노쇼의 VisitRecord 기록여부, 전화번호 유일키 강제여부(가족공유), 퇴사직원 과거실적 인정여부, 마이너스재고 허용여부, 예약 동시진행(중복) 허용범위 — 5영역 전부 최소 1건씩 즉시결정 대상에 걸려있음. 결정만 정리, 코드/컬럼 추가 없음(2026-06-23).

## 3-8. 예약 시간중복 정책 — 업종별(헤어/네일/마츠게) 상세분석

`BOOKING_OVERLAP_POLICY_ANALYSIS.md` — §1-7(예약 동시진행 허용범위) 심화분석. **결론**: 현재 `overlaps()`(전체중복=충돌)는 네일·마츠게에는 정확하고 헤어에는 과도하게 엄격함. 마츠게는 베드(좌석)수가 담당자보다 먼저 막히는 진짜 병목인데 좌석 엔티티 자체가 없어 과잉예약을 시스템이 못 잡음. **권장**: 컬럼추가 없이 "정책을 선택하는 한 지점"만 A-3 설계에 구조적으로 분리해두고(지금은 전업종 엄격정책 유지), 좌석/베드 엔티티·업종분류 필드·구간별시간분리는 별도 제안목록으로 분리(신규엔티티 필요, 추가하지 않음). 설계 검토만 수행(2026-06-23).

## 3-9. A-1(recordVisit 연결) 구현 전 최종 점검

`A1_PREFLIGHT_REVIEW.md` — A-1 착수 전 점검. 핵심 트리거 지점은 `PaymentRepository.pay()`의 기존 `newStatus='completed'` 분기(이미 초과결제 차단 로직이 중복호출 방지장치로 작동함을 코드로 재확인). **가장 시급한 구조적 리스크**: `Order`에 `bookingId` 컬럼이 없어 분할결제 중 화면이 직접 들고다녀야 하고, 중간에 잃어버리면 `completeBooking()` 호출이 조용히 누락됨. 0원주문(§1-2)·환불처리(§2-2)는 결정을 미뤄도 "미기록/비침범" 방향으로 범위를 좁게 유지 가능. **구현착수 전 1순위 결정**: 이번 A-1에 예약경로(completeBooking 연동)를 포함할지, 워크인 경로만 먼저 출시할지. 검토만 수행, 코드 없음(2026-06-23).

## 3-10. A-4(직원 재직/퇴사 상태) 구현 전 분석 — 컬럼 추가 없이

`A4_PREFLIGHT_REVIEW.md` — 컬럼 추가 금지 조건에서 A-4 재검토. **핵심 발견**: `Staff.accountStatus`가 이미 nullable TEXT라 신규 컬럼 없이 4번째 값(예: '退職済み') 추가만으로 재직/퇴사 상태 표현 가능 — 마이그레이션 불필요. `removeStaff()`를 `_db.delete()`→상태갱신으로 교체하면 `Shift`의 cascade 삭제 문제도 부수적으로 해소됨. A-1/A-2는 Staff 상태를 조회하지 않는 구조라 회귀 없음(코드 변경 불필요). **부족한 부분**: `watchStaff()` 필터링 책임 소재 없음, 퇴사자 전화번호가 신규등록 중복체크를 막는 새 부작용, `_assertStaffAvailable()`이 퇴사자에게도 예약을 계속 허용. 마이그레이션 불필요. 설계 검토만 수행(2026-06-23).

## 3-11. A-5(재고 이력 보존) 구현 전 프리플라이트 검토

`A5_PREFLIGHT_REVIEW.md` — A-4와 달리 `InventoryItem`엔 재활용 가능한 여유 필드가 없어, "기존 컬럼에 새 값 추가" 전략을 그대로 적용할 수 없음을 확인. **권장 방향**: `deleteItem()`을 상태전환이 아니라 "InventoryLog가 1건 이상 있으면 삭제 자체를 거부"하는 정책으로 — 컬럼 추가 없이도 이력 영구보존 가능. **추가 발견**: `adjustQuantity()`는 `cancelOrder()`와 달리 트랜잭션으로 안 묶여있어 동시결제 시 lost update 위험(현재는 결제-재고 연동 자체가 없어 표면화 안 됨, F-INV-00 재확인). "현재수량(저장값)"과 "이력합산값"이 서로 다른 소스라 정합성 검증이 전혀 없다는 점도 식별 — 재고조사(棚卸) 구조화와 함께 본 A-5 범위 밖으로 분리. 설계 검토만 수행(2026-06-23).

## 3-12. A-3(예약 변경) 구현 전 프리플라이트 검토

`A3_PREFLIGHT_REVIEW.md` — MVP 5건 중 영향범위 최대 항목 착수 전 점검. **핵심 결론**: `Booking.status`에 별도 "변경됨" 상태 불필요(`'confirmed'` 유지하며 필드만 갱신). 가장 직접적인 위험은 `_assertStaffAvailable()`을 그대로 재사용하면 자기 자신과 항상 충돌판정되어 모든 변경이 거부되는 것(자기제외 로직 필수). 조용한 위험은 `updateBooking()`이 `cancelBooking()`/`completeBooking()`과 같은 종결상태 가드를 안 쓰면 이미 완료/취소된 예약도 변경 가능해져 A1_A2_BOUNDARY 원칙이 깨지는 것. 변경 이력은 테이블 추가 금지로 overwrite 방식이 유일한 선택(기존 cancelBooking도 같은 한계). 워크인↔예약 전환은 데이터 모델 자체가 표현 못해 범위 제외. A-1/A-2/A-4/A-5에는 영향 없음(Booking↔Order FK 부재로 구조적 독립). 설계 검토만 수행(2026-06-23).

## 3-13. A-4×A-5 통합 검토 + 시스템 연결 관점

`A4_A5_INTEGRATED_REVIEW.md` — A-1~A-3 구현 완료 시점의 실제 데이터흐름 위에서 A-4/A-5를 재검토. **신규 발견**: accountStatus 재활용은 이원화 필요(`連結済み`만 상태전환, `null`/`待機中`은 하드삭제 유지 — 보존할 이력이 없으므로). `_assertStaffAvailable()`이 퇴사자에게도 신규예약을 계속 허용하면 A-4가 도입돼도 무력화됨(A-4와 한 묶음으로 결정 필요). **시스템 연결 관점의 핵심**: Inventory↔Product 분리(F-INV-00)는 "의도된 업무경계"라 유지가 맞지만, Staff↔Booking/Payment/VisitRecord의 느슨한 텍스트참조는 "기술적 단순화가 누적된 미해결 부채"로 성격이 다름 — 같은 수준의 분리원칙으로 취급하면 안 됨. Product↔InventoryItem은 공유키 자체가 없어 결제-재고 연동이 구조적으로 불가능(테이블/컬럼 추가 금지와 정면충돌, A-6 이후로 명시적 분리). 설계 검토만 수행(2026-06-23).

## 3-14. A-4×A-5 결제흐름 기준 통합검증 (분석→결정 단계)

`A4_A5_PAYMENT_FLOW_VERIFICATION.md` — §3-13 분석을 실제 pay() 흐름 위에서 결정 수준으로 확정. **연결해야 하는 영역(유일)**: `_assertStaffAvailable()`에 퇴사직원 신규예약 차단 검증 — A-4와 같은 작업단위로. **절대 연결하면 안 되는 영역**: pay()/recordVisit() 경로에 Staff 상태검증 추가(이미 끝난 시술의 결제가 직원상태변경 때문에 실패하는 시간차 오류 발생), Product↔InventoryItem 임시 매핑(이름매칭 등) 우회연결. **1순위 위험**: adjustQuantity() 비원자성 — 매핑구조(A-6+)가 생기기 *전에* 먼저 해결해야 할 선행조건. visit기준/payment기준은 별개 질문이 아니라 이미 같은 지점(A1_A2_BOUNDARY 트리거)임을 재확인. 설계 검토만 수행(2026-06-24).
