# salon-pos v3 정의서 — 개요

> **목적**: 이 문서 세트(v3)만 보고 일본 살롱 POS를 처음부터 구현해도 토스플레이스(TossPlace) 100% 벤치마킹 기준에서 벗어나지 않는 품질로 만들 수 있도록, 화면이 아니라 **자료조사(`../toss_benchmarking.md`) 기반의 정의서**로 다시 작성한다.
> **베이스 자료**: `design/spec/toss_benchmarking.md`(토스 실제 화면 검증 기록) + `design/mockups/v2/ja_toss100/`(기존 구현 목업, 1차 참고용이며 정의서가 우선)
> **작성 시작**: 2026-06-23

---

## 1. 문서 구성 원칙

영역(고객관리, 선불권 등) 하나당 항상 3종 문서를 세트로 만든다.

| 문서 | 파일명 | 내용 |
|---|---|---|
| 기능 정의서 (Feature Spec) | `feature_spec.md` | 기능이 "무엇을, 왜, 어떤 규칙으로" 동작하는지. 비즈니스 규칙·예외처리·제약조건 중심. 화면 레이아웃은 다루지 않음. |
| 화면 정의서 (Screen Spec) | `screen_spec.md` | 화면 번호·Zone 구조·UI 요소·상태분기·인터랙션 흐름. "어떻게 보이고 어떻게 동작하는지"를 화면 단위로. |
| 데이터 정의서 (Data Spec) | `data_spec.md` | 엔티티·필드·관계·산출 로직(pseudo-code). 기능정의서의 규칙을 실제 저장 구조로 변환. |

세 문서는 서로를 참조한다 — 기능정의서의 규칙 ID(예: `F-CUST-01`)를 화면정의서·데이터정의서에서도 그대로 인용해 추적 가능하게 한다.

## 3. 현지화 원칙 (필수)

`[[project_salon_pos_v2_principles]]` 메모리 참조 — 모든 화면 정의서의 UI 라벨은 **일본어 원문**으로 표기하고, 필요시 한국어 괄호 설명을 보조로 붙인다. 화폐는 ¥, 날짜는 年月日, 단위는 件/名/分/種을 사용한다. 본 문서(기능/데이터 정의서)는 작업 편의상 한국어 위주로 작성하되, 사용자에게 노출되는 모든 문자열(라벨/버튼/안내문구)은 일본어로 명시한다.

## 3-1. 전체 화면×ERD 통합 문서

`SCREENS_ERD_TABLES.md` — 27개 화면 전체 기능목록 + 현재 구현된 21개 테이블 기준 ERD/테이블정의 + 부족한 엔티티 제안(Branch/OptionGroup/Receipt/AuthSession) + 샘플데이터. 코드/SQL 미생성, 분석 문서로만 작성(2026-06-23).

## 3-2. 설계 리뷰 (실제 운영 업무 기준)

`DESIGN_REVIEW.md` — `SCREENS_ERD_TABLES.md` 기준으로 예약/고객/재고/결제/직원 5영역의 실제 운영 프로세스 충족 여부를 레포지토리 로직까지 재검토. **최우선 발견**: 방문완료 시점에 `CustomerRepository.recordVisit()`을 호출하는 코드가 어디에도 없어, 고객 그룹 자동분류(F-CUST-01)가 화면상으로만 존재하고 실제로는 작동하지 않는 상태. 테이블 변경 없이 리뷰만 수행(2026-06-23).

## 3-3. 우선순위 분류 (출시 단계별)

`PRIORITIZATION.md` — DESIGN_REVIEW.md의 발견사항을 MVP 필수/베타 추가가능/정식출시 이후로 분류, 항목별 이유·예상영향도·구현난이도 정리. MVP 필수 5건(방문완료 연동·예약변경·직원 재직상태·재고이력보존 등) 중 2건(recordVisit 연결, 방문완료 상태전환)은 난이도 "하"로 기존 메서드 호출지점만 정하면 해결 가능. 분류만 수행, 설계 변경 없음(2026-06-23).

## 3-4. MVP 필수 5건 영향범위 맵

`MVP_IMPACT_MAP.md` — PRIORITIZATION.md §A 5건 각각의 영향 화면/테이블/Repository를 매핑하고 구현순서를 제안. 권장순서: A-2(예약완료상태)→A-4(직원재직상태)→A-1(recordVisit연결)→A-5(재고이력보존)→A-3(예약변경, 영향범위 최대라 마지막). 설계 문서만 작성, 코드 없음(2026-06-23).

## 3-5. A-2 구현 전 최종 점검

`A2_PREFLIGHT_REVIEW.md` — A-2(예약 방문완료 상태전환) 착수 전 점검. 영향범위는 `Booking.status` 단일 컬럼·`BookingRepository` 단일 클래스로 매우 좁음. **선행 의사결정 필요**: A-1(결제경로 트리거)과의 호출 순서를 먼저 정하지 않으면 VisitRecord 이중 적재 위험. 예외케이스 6건 중 "미래예약 완료처리 차단 여부"·"예약금 필드 비침범"은 구현 전 결정 필요. 검토만 수행, 코드/컬럼 추가 없음(2026-06-23).

## 3-6. A-1×A-2 업무 경계 정의

`A1_A2_BOUNDARY.md` — 두 항목이 한 트리거(결제완료)에 종속되도록 경계 확정. **방문확정 = `Order.status→'completed'` 단일 트리거**(예약 유무 무관), `recordVisit()` 호출책임은 `PaymentRepository.pay()` 단일 지점으로 한정(BookingRepository/화면이 직접 호출 금지). `completeBooking()`은 bookingId가 전달된 경우에만 같은 트리거에서 부수 호출. 경계 밖으로 남긴 케이스: 0원 주문(전액할인)이 영원히 완결 안 되는 문제, 환불 시 VisitRecord 처리방식. 설계 경계만 정의, 코드 없음(2026-06-23).
