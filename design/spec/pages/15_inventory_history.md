# 15 · 在庫変動履歴 — 화면 정의서

> **파일**: `design/mockups/v2/ja/15_inventory_history.html`
> **목적**: 재고 변동 이력을 카테고리·날짜·변경이유별로 조회.
> **벤치마킹**: **토스포스 100% 기반** — `01_tosspos_analysis.md` §4-6 "재고관리 히스토리: 카테고리별, 날짜별, 변경이유별 필터" 그대로 구현. 별도의 "입고/출고/발주" 화면은 토스에 없으므로 만들지 않음(아래 §3 참조).
> **진입**: `14_inventory_list` 「変動履歴を見る」
> **이동**: 戻る → `14_inventory_list`

## 1. 구역 구성

| 구역 | 역할 |
|------|------|
| Zone B | 戻る + 날짜 네비게이션(‹›今日) + 카테고리 select + 변경이유 select |
| Zone C | 테이블: 시각/상품명/이유배지/변동(±)/변경후재고/담당자 |
| Zone D | 본일 변동 요약(입荷/판매소비/수동조정/폐기손실 건수), 이유 범례 |

## 2. 변경이유(4종)

| 이유 | 의미 | 색상 |
|------|------|------|
| 入荷 | 발주 없이 단순 "입고 받음" 기록(§3 참조) | 초록 |
| 販売消費 | POS 판매 시 자동 차감 | 파랑 |
| 手動調整 | 14의 +/- 빠른수정으로 인한 변경 | 보라 |
| 廃棄・損失 | 파손/유효기간 만료 등 | 빨강 |

## 3. 16(발주관리) 화면을 만들지 않은 이유

토스포스 분석 자료(`01_tosspos_analysis.md`)에는 외부 공급업체 대상 "발주(주문)" 플로우가 없음 — 재고관리는 §4-6의 "재고 수량 수정"(+버튼으로 직접 입고분 반영)이 전부. **토스 100% 기반** 원칙에 따라 가상의 발주관리 화면을 새로 만들지 않고, 입고는 14의 빠른수정(+버튼) 또는 본 화면의 "入荷" 이유로 기록되는 것으로 흡수함. INDEX.md에도 동일하게 명시.
